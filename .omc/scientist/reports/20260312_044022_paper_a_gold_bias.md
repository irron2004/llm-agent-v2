# Paper A Eval Set: Gold Label Bias Analysis
**Date:** 2026-03-12
**Analyst:** Scientist (claude-sonnet-4-6)

---

## [OBJECTIVE]
Quantify gold label bias in Paper A's eval set (`query_gold_master_v0_5.jsonl`, 472 queries) and Phase3 new gold (`phase3_new_gold.json`, 108 queries) to assess whether the evaluation design fairly measures scope-aware retrieval (device-level filtering).

---

## [DATA]
- **v0_5 eval set:** 472 queries, 22 fields per query
  - Splits: dev=421, test=51
  - scope_observability: implicit=150, explicit_device=146, explicit_equip=96, ambiguous=80
  - Sources: 11 distinct sources (sop79_csv, synthetic_*, golden_set_*)
  - Queries with gold_doc_ids > 0: 399/472 (84.5%); zero-gold: 73 (15.5%)
- **Phase3 new gold:** 108 queries (TREC-pooled, LLM-judged)
  - Balanced across obs types: ambiguous=30, implicit=29, explicit_device=28, explicit_equip=21

---

## ANALYSIS 1: v0_5 Gold Structure

### [FINDING 1] Extreme gold set collapse — 98.5% of queries share their exact gold set
Within the 342 queries that have gold docs AND a canonical device name, only 5 queries (1.5%) have a gold set not shared by any other query. The remaining 337 queries (98.5%) share their exact (device, gold_doc_ids) tuple with at least one other query.
[STAT:n] n=342 (queries with device + gold)
[STAT:effect_size] 28 unique (device, gold_set) combos covering 342 queries → 12.2x oversampling on average
[STAT:effect_size] Median gold set: shared by 11.7 queries; mean by 12.2 queries

### [FINDING 2] SUPRA_XP dominance creates a single-point-of-failure in 34.1% of the eval set
161 queries (34.1% of all 472) — including all 158 sop79_csv/sop79_csv_masked queries — share the identical gold set: 5 SUPRA_XP SOP documents. One retrieval decision (can the model find these 5 docs?) determines recall@5 for a third of the entire benchmark.
[STAT:n] n=161 queries with identical gold
[STAT:effect_size] 161 queries → 1 unique gold set (161:1 concentration ratio)
[STAT:effect_size] SUPRA_XP has 16.5x more queries per unique gold doc than the 1:1 baseline

### [FINDING 3] Gold sharing rate is near-maximal: mean=0.989, median=1.0
For each query, the "sharing rate" = fraction of its gold docs also used by another query for the same device. 98.8% of queries have sharing_rate = 1.0 (ALL gold is shared).
[STAT:n] n=342
[STAT:effect_size] Mean sharing rate = 0.989 (out of 1.0 maximum)
[STAT:effect_size] Queries with entirely unique gold: 3/342 (0.9%)

### [FINDING 4] Scope filter marginal value is approximately zero for 71.2% of queries
71.2% of valid queries (284/399) have ALL their gold docs named with device-encoded IDs (global_sop_{device}_*). For these queries, a no-filter retrieval system can find the correct docs through keyword overlap alone — the device name in the query already indexes the docs. Scope filtering provides no additional recall benefit.
[STAT:n] n=399 (queries with gold)
[STAT:effect_size] explicit_device: 95.9% have all-device-indexed gold
[STAT:effect_size] implicit: 88.6% have all-device-indexed gold
[STAT:effect_size] explicit_equip: 0.0% have any device-indexed gold (all GCB_* docs)
[STAT:effect_size] ambiguous: 48.8% have all-device-indexed gold

### [FINDING 5] "Document → Question" construction creates inherent recall inflation
sop79_csv source: 79 explicit_device + 79 implicit = 158 queries, all SUPRA_XP, ALL with n_gold=5, ALL with device-indexed gold docs, ALL sharing the same single gold set. These queries were generated by reading the source SOP documents and asking questions about them. Any retrieval system that returns the source document achieves recall=1 for all 158 queries simultaneously.
[STAT:n] n=158 (sop79 source), n=472 (full set)
[STAT:effect_size] sop79 constitutes 33.5% of total eval queries (158/472)
[STAT:effect_size] 100% of sop79 queries have n_gold=5 and identical gold set

## ANALYSIS 2: Phase3 New Gold

### [FINDING 6] Phase3 strict gold is sparse: only 33.3% of queries have any strict gold
After TREC pooling + LLM judging, 72/108 queries (66.7%) have zero strict gold docs. Only 36 queries have ≥1 strict gold (mean=0.5 per query across all 108). Loose gold is more abundant (mean=3.7 per query) but present for only 88.9% of queries.
[STAT:n] n=108 (phase3 queries)
[STAT:effect_size] Strict gold present: 36/108 (33.3%)
[STAT:effect_size] explicit_equip has the lowest strict gold rate: 9.5% (2/21 queries)

### [FINDING 7] Phase3 loose gold is predominantly cross-device for explicit_equip queries
For explicit_equip queries (n=21), the gold docs are heavily cross-device: 50 cross-device loose gold docs vs only 5 in-scope. This means a device-level scope filter applied to explicit_equip queries would eliminate 90.9% of valid gold docs, actively hurting recall.
[STAT:n] n=21 explicit_equip queries
[STAT:effect_size] Cross-device loose gold: 50 docs (90.9% of identifiable gold)
[STAT:effect_size] In-scope loose gold: 5 docs (9.1% of identifiable gold)

### [FINDING 8] Phase3 also biased toward no-filter systems via pooling construction
The phase3 pool was constructed from B0~P1 retrieval results (predominantly no-filter systems). Gold docs discovered by filter-only systems are systematically absent. This means scope-aware recall measured against phase3 gold underestimates the true benefit of scope filtering.
[STAT:n] n=108 (phase3 queries)
[STAT:effect_size] Ambiguous queries show highest strict gold rate (43.3%) — consistent with no-filter advantage in pooling
[STAT:effect_size] explicit_equip has lowest strict gold rate (9.5%) — scope filter results were likely underrepresented in pool

## [LIMITATION]
1. **Device extraction from doc_id is heuristic.** The `extract_device_from_doc_id` function uses string prefix matching; edge cases (e.g., GENEVA_XP_ADJ, SUPRA_N_SERIES) may be misclassified as cross-device when they are actually in-scope variants.
2. **"Marginal value = 0" claim assumes keyword overlap is sufficient.** In practice, device-indexed docs with `global_sop_supra_xp_*` names will score highly in BM25 when the query contains "SUPRA XP". Dense retrieval may behave differently.
3. **Phase3 cross-device analysis limited to explicit queries with device field set.** 57 phase3 queries (ambiguous + some implicit) have no device field, so their alignment could not be assessed.
4. **The sop79 recall inflation is an upper bound.** The actual inflation depends on retrieval rank position — if the 5 gold docs appear outside top-k, the bias is reduced.
5. **Sharing rate computed only for queries with non-empty canonical_device_name.** 107 queries with empty device (mostly equip-centric and ambiguous) were excluded from this analysis.
