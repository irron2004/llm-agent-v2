Stage 1 — Eventizer

Raw 센서 시계열을 "정비 가능한 증상"으로 변환한다

입력: 원시 센서값
sensor window
apc_position_actual: 91.2%
apc_position_setpoint: 55.0%
tracking_error: 36.2%
duration: 42초
pressure_oscillation: 0.17Hz
actuator_current: 정상
calibration_age: 78일
출력: 구조화된 이벤트
event node
component: APC
symptoms:
  - tracking_failure  0.93
  - high_saturation   0.81
  - pressure_oscillation 0.72
context:
  recipe_step: pressure_stabilization
  calibration: overdue
핵심: 문서는 anomaly가 아니라 symptom / failure mode / action 수준에서 연결된다. 그래서 raw 센서값이 아니라 증상 수준으로 올려야 문서와 연결 가능하다.

Stage 2 — Temporal Uncertainty Alignment

정비 로그의 시간은 실제 고장 시점과 다르다

Gold Link
원인, 부품, 시각이
로그에 명시
확실
Silver Link
같은 component +
인접 시각
가능성 높음

같은 asset의
maintenance episode
약한 연결
이 모듈이 Paper D의 가장 강한 novelty 포인트. 기존 연구는 이 문제를 전처리 수준으로만 다룬다.

Stage 3+4 — Maintenance Case + Document Retrieval

현재 이벤트와 유사한 과거 사례를 찾고, 관련 SOP/Manual을 검색한다

Case Retriever
역할: 과거 정비 사례 top-k 검색

학습 방법:
• Sensor encoder Es(window)
• Log encoder Em(maintenance)
• Contrastive loss로 공통 공간 정렬

Hard Negative:
• 같은 부품, 다른 증상
• 같은 증상, 다른 장비
• 같은 action, 다른 failure mode
Document Retriever
역할: SOP/Manual/FMEA 검색

검색 구성:
Score = 0.35 × dense
     + 0.25 × BM25
     + 0.20 × metadata
     + 0.20 × graph

왜 분리?
• Case = 과거 실제 행동 근거
• SOP = 공식 절차 근거
Ltotal = Lretrieval + λ1 Lfault_type + λ2 Ltime
contrastive retrieval + failure mode 분류 + temporal compatibility를 동시 학습

Stage 5+6 — Grounded Diagnosis

LLM은 "지식 생성기"가 아니라 "근거 편집기"로 사용한다

최종 답변 예시
가능한 원인 (confidence: 0.78)
  1. APC calibration drift
  2. Valve sticking
  3. Actuator feedback mismatch

근거 문서
  - SOP_APC_013 §4.2.1: setpoint 추종 실패 시 calibration 우선
  - MANUAL_APC_007 §3.4: feedback mismatch 시 consistency 점검
  - CASE_2025_11_03: 유사 증상에서 linkage sticking 확인

점검 순서
  1) manual jog test 수행
  2) actual vs feedback consistency 확인
  3) calibration procedure 실행

안전 주의
  - auto pressure control 불안정 중 recipe 지속 금지
  - shutdown/override는 SOP 절차 준수 후 실행

"지도 만들기" vs "길 찾기"

Paper D는 두 가지 일을 한다

🗺️
지도 만들기 (Map Building)
• 센서 데이터 → event node 정의
• 정비 이력 → case node 정의
• SOP/FMEA → document node 정의
• 노드 간 relation/edge 정의
• Ontology 구성

sensor → component → symptom
→ failure mode → action → document
🧭
길 찾기 (Path Finding)
• 새로운 이상이 들어오면
• 지도 위에서 유사 state 검색
• 연결된 maintenance case 회수
• 관련 SOP/Manual 검색
• Grounded diagnosis 생성

current event → similar past events
→ linked cases → linked docs → diagnosis
Graph-based anomaly 아이디어 = 지도를 더 잘 만드는 방법
Paper D의 핵심 기여 = 그 지도 위에서의 retrieval + grounded diagnosis

첫 번째 실증: ES 조회 결과

실제 데이터에서 센서 → 문서 연결이 성립하는가?

센서 문서 Hit 문서 예시 Pilot 적합성
APC_Position 25 "root cause of APC position increase" 적합
APC_Pressure 58 "pressure hunting of the APC valve" 적합
Temp1 80 "Temp1 trigger temperature differential" 적합
Temp2 242 "intermittent Temp2 interlock alarm" 적합
APC_SetPoint 0 — 매핑 필요
EPD_*, Gas_*, SourcePwr* 0~1 — 사전 구축 후
62개 센서 중 7개(11%)만 raw name으로 직접 연결됨
→ 이것 자체가 sensor-document semantic gap의 실증 근거
→ APC_Position + APC_Pressure로 pilot 시작 가능
