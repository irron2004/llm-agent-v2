# HVSR 제안 검토 의견

**작성일**: 2026-03-18
**대상**: HVSR (Hypothesis-Verified Scope Retrieval) 제안서
**결론**: 진단은 정확하나, 알고리즘이 과설계됨. 간소화 대안 제시.

---

## 1. 진단 평가: 정확함

> "cross-equipment contamination은 재랭킹 문제가 아니라 candidate generation / scope selection 문제다"

이전 분석에서 확인된 근거와 완전히 일치한다:

- P6/P7 candidate ceiling: B3 recall@10 = 60.7%이므로 39.3%의 쿼리에서 gold가 top-10에 없음
- λ convergence paradox: λ < 0.5는 무효과, λ ≥ 0.9은 hard filter와 동일 → 효과적 중간 지대 없음
- B4 vs P6 per-query: B4만 성공 157건, P6만 성공 0건 — 전부 후보집합 문제

**"P6/P7을 negative result로 두고, 새 알고리즘의 동기로 위치시키자"는 방향도 적절하다.**

## 2. HVSR 구조적 문제

### 2.1 하이퍼파라미터 폭발 (Overfitting Risk)

| Stage | 파라미터 | 개수 |
|-------|---------|------|
| Stage 1 (Proposal) | λ_a, λ_l, λ_c, λ_p, M, γ, ε | 7 |
| Stage 3 (Verification) | α, β, δ, ξ | 4 |
| Stage 4 (Selection) | η | 1 |
| SSG | B (shared cap), gate threshold | 2 |
| **합계** | | **14+** |

평가 쿼리 578개(explicit_equip은 149개)로 14개 파라미터를 튜닝하면 과적합 위험이 높다.
학술 논문에서 "파라미터 14개, 데이터 578개"는 리뷰어의 즉각적 지적 대상이다.

### 2.2 레이턴시 비용

```
현재 B4:  1× retrieval (단일 hard filter)
HVSR:     1× probe + M× hypothesis retrieval = 4~6× retrieval (M=3~5)
```

현재 `auto_parse_node`가 규칙 기반(LLM 호출 없음)인 이유가 "속도 최적화"이다
(`langgraph_agent.py:4728` 주석: "규칙 기반만 사용 (LLM 호출 없음 - 속도 최적화)").
생산 PE 시스템에서 4~6× 레이턴시 증가는 실용성을 크게 떨어뜨린다.

### 2.3 Lexical Device Router (S_lex)의 실효성 의문

반도체 PE 도메인 특성상 장비 간 용어 중복이 심하다:

- 공통 용어: PM, chamber, wafer, slot valve, recipe, gas line, MFC, throttle valve...
- SOP 구조: 장비 간 동일 포맷, 유사 절차
- 장비 고유 키워드(SUPRA, PRECIA, INTEGER 등)는 이미 alias 매핑으로 처리됨

S_lex가 alias 이상의 식별력을 제공하려면 27장비 × 2 doc-type group의 term probability
분포를 구축해야 하나, 비용 대비 효과가 불분명하다.

### 2.4 Probe Retrieval의 자체 모순

Stage 1에서 probe retrieval로 device 분포를 관찰한다면,
그 probe 결과의 top device를 바로 사용하면 Stage 2~4가 불필요해진다.

"probe로 이미 답을 얻었는데 왜 다시 M번 검색하는가?"라는
알고리즘 내재적 모순이 있다.

## 3. 실제 이득 구간 분석

| 쿼리 유형 | 수량 | 현재 B4 | HVSR 추가 이득? |
|-----------|------|---------|----------------|
| explicit_device (429) | 74.2% | parser 정확 → B4 이미 강력 | **거의 없음** |
| explicit_equip (149) | 25.8% | parser 0% → B4 불가 | **이론적 이득**, 단 equip→family 매핑으로 더 간단히 해결 가능 |
| implicit (v0.7, 578) | — | sticky policy로 일부 커버 | **cold-start에서만** |

HVSR의 5-stage 파이프라인이 실질적 이득을 보는 구간은
**explicit_equip 149건 + implicit cold-start** 뿐이다.
이 구간을 위해 전체 파이프라인을 4~6× 느리게 만드는 것은 비효율적이다.

## 4. 더 간단한 대안이 이미 존재함

### 4.1 ES device_boost (코드베이스에 구현 완료, 미평가)

`backend/llm_infrastructure/retrieval/engines/es_search.py` L514-540:

```python
# _build_text_query() 내부
"should": [{
    "term": {"device_name": {"value": device_boost, "boost": device_boost_weight}}
}]
```

- **pre-retrieval** 단계에서 후보집합 자체를 바꿈 (P6/P7의 candidate ceiling 해결)
- hard filter보다 유연 (다른 장비 문서도 점수 낮게 포함 가능)
- **단일 ES 쿼리**로 HVSR Stage 2와 동일한 효과
- 기본값 `device_boost_weight=2.0`, 파라미터 1개

### 4.2 Family-Level Filter

```
if device_name detected → filter by device_name          (B4)
elif equip_type detected → filter by equipment_family    (신규)
else                     → no filter / device_boost      (fallback)
```

- explicit_equip 149건을 직접 해결
- HVSR의 lexical router보다 결정적(deterministic), 파라미터 0개
- 근거: Stoica et al. (2007) hierarchical faceted metadata
- 예상 효과: PRECIA→INTEGER plus 등 cross-family 오염(173건) 제거

### 4.3 shared_cap (B4.5 수정)

- B4.5 실패 주원인: shared_overload 85.7% (`2026-03-14_b45_failure_decomposition.md`)
- `shared_cap=2` 같은 단순 quota로 SSG와 동일한 효과
- SSG의 g_shared(q) 분류기 불필요

## 5. 학술적 신규성 우려

HVSR의 multi-hypothesis retrieval은 **federated/selective search**
(Shokouhi & Si, 2011)와 동일 구조이다 — 이미 논문에서 인용한 선행 연구.

"device별 독립 retrieval → 결과 비교 → 최적 선택"은
collection selection의 표준 패러다임이며, 차별화 포인트가 부족하다.

Stage 3의 verification scoring(Mass, Coverage, DocTypeMatch)은
ad-hoc feature engineering으로, 이론적 근거가 약하다.

---

## 6. "B4 = oracle"이라는 전제 오류

제안서에서 "B4: oracle hard filter upper bound"로 표현하지만,
현재 B4는 **parser가 감지한 device**를 사용한다.

- explicit_device: parser 정확도 높음 → oracle에 근접
- explicit_equip: parser 0% → B4 자체가 작동하지 않음
- 진짜 oracle: gold label의 device를 사용하는 것

이 구분이 모호하면 HVSR의 upper bound 주장이 흔들린다.

## 7. 제안: 실현 가능한 간소화 대안

HVSR의 핵심 아이디어를 살리되 복잡도를 1/10으로 줄인 구조:

### "Hierarchical Scope Routing" (HSR)

```
Stage 1: Multi-signal device detection (현재 구현 확장)
  ├── alias/regex parser        (기존, langgraph_agent.py L4730)
  ├── equip_type → device_family 매핑  (신규, 파라미터 0)
  └── sticky context inheritance       (기존, langgraph_agent.py L4746)

Stage 2: Confidence-gated retrieval
  ├── HIGH (device 감지됨)    → hard filter (B4)
  ├── MID  (family만 감지)   → family-level filter (신규)
  └── LOW  (미감지)          → device_boost soft retrieval (기존 코드)

Stage 3: Shared quota
  └── shared_cap = 2 (B4.5 수정)
```

### HSR vs HVSR 비교

| 항목 | HVSR | HSR (간소화) |
|------|------|-------------|
| 파라미터 수 | 14+ | 2 (shared_cap, device_boost_weight) |
| 레이턴시 | 4~6× | 1× |
| 신규 코드 | 5-stage 파이프라인 전체 | family 매핑 + shared_cap |
| 기존 코드 활용 | 거의 없음 | device_boost, sticky policy 전부 활용 |
| explicit_equip 해결 | lexical router (불확실) | family-level filter (결정적) |
| 과적합 위험 | 높음 (14개/578쿼리) | 낮음 (2개/578쿼리) |
| 논문 기여 | "multi-hypothesis scope retrieval" | "hierarchical scope routing with confidence gating" |

## 8. 결론

### 동의하는 부분
- P6/P7 실패는 구조적 한계이며, 이를 negative result + 새 알고리즘 동기로 활용하는 방향
- scope control은 pre-retrieval에서 수행해야 한다는 핵심 메시지
- shared 문서를 별도로 제어해야 한다는 방향 (SSG의 기본 아이디어)
- 논문 기여를 "hard filter 좋음/soft 나쁨"에서 "알고리즘 제안"으로 격상하자는 제안

### 동의하지 않는 부분
- 5-stage 파이프라인의 복잡도가 문제 규모에 비해 과도함
- Lexical router를 raw count로만 두면 실효성이 약함 (log-odds/Bayesian likelihood 필요)
- Probe retrieval은 단독 신호가 아니라 proposal 보정 신호로 제한해야 함
- 14개 하이퍼파라미터를 578쿼리로 튜닝할 수 있다는 전제

### 최종 권고

1. **HVSR의 진단과 방향성은 채택**하되, 실현체는 HSR(간소화 버전)으로 대체
2. HVSR 전체는 **future work**으로 언급 ("multi-hypothesis extension")
3. 즉시 실험 가능한 항목: (A) device_boost 평가, (B) family-level filter, (C) shared_cap
4. 이 세 가지만으로도 논문의 알고리즘 기여를 충분히 확보할 수 있음

## 9. 추가 반영: Device 후보 생성 기준 (Bayesian lexical router)

요청 반영 요지:

- "장비별 용어 count"는 맞는 방향이지만 raw count 그대로가 아니라
  **Bayesian keyword likelihood / log-odds**로 정식화해야 함
- hard evidence가 있으면 후보를 좁히고, 없으면 top-M hypothesis를 여는 구조가 타당

### 9.1 Hard evidence 우선

아래 신호가 있으면 후보를 사실상 고정:

- explicit device mention
- alias canonicalization
  - `ZEDIUS XP -> SUPRA_XP`
  - `SUPRA V -> SUPRA_VPLUS`
- explicit equip_id mention (equip->device mapping 가능 시)

운영:

- hard evidence 존재 시 `M=1~2`
- 부재 시 `M=3~5`

### 9.2 Bayesian lexical score

문서군 `c`(procedure/history)를 구분해 device 점수 계산:

`Score(d|q,c) = log P(d) + sum_t tf(t,q) * log(P(t|d,c)+eps)`

안정화 버전(log-odds):

`Score(d|q,c) = sum_t tf(t,q) * log((P(t|d,c)+eps)/(P(t|bg,c)+eps))`

의미:

- 공통 단어는 약화
- 장비 특이 단어는 강화

### 9.3 Context + probe 보정

context bonus:

`Score(d|q,h) = Score_lex(d|q) + gamma * 1[d=d_{t-1}]`

probe 보정:

`S_probe(d|q) = log(P_r(d|q)+eps)`

최종 proposal:

`S_prop(d)=w_alias*S_alias + w_lex*S_lex + w_ctx*S_ctx + w_probe*S_probe`

후보집합:

- `TopM` 고정 또는 posterior threshold/margin rule 사용

## 10. 기존 draft 대비 개선점 (명시)

기존 draft 구조:

1. regex/dictionary parser
2. sticky context inheritance
3. 단일 device hard filter
4. soft scoring(P6/P7)

개선 구조(HVSR 방향):

1. single parse -> **multi-hypothesis proposal(top-M)**
2. parser-only -> **Bayesian lexical + context + probe evidence**
3. immediate filter -> **가설별 retrieval + verification**
4. naive shared allowance -> **selective shared gate/shared cap**

한 줄 요약:

> 기존은 "한 번 파싱해서 맞으면 필터"였고,  
> 개선안은 "Bayesian하게 top-M scope를 제안하고 retrieval 증거로 최종 scope를 선택"한다.
