# page1 handoff guide

이 문서는 `data_specification.md`를 바탕으로 공식 양식 page1/page2에 내용을 옮겨 적을 때,
동료가 빠르게 맥락을 이해하고 수정할 수 있도록 만든 보조 문서다.

> 핵심 원칙
> - **공식 양식에 들어가는 최종 문안**은 `data_specification.md`를 따른다.
> - 이 문서는 그 문안의 **배경, 근거, 수정 포인트, 주의사항**을 설명한다.
> - page2는 page1의 우측 페이지이므로, 실질적으로는 같은 양식의 연장선으로 본다.

---

## 1. 이 문서의 역할

이 문서는 요약본이 아니라 **전달용 설명서**다. 동료가 아래 상황에서 참고하도록 설계했다.

1. 왜 이런 문구를 썼는지 알고 싶을 때
2. 어떤 숫자가 원본 기준이고 어떤 숫자가 chunk 기준인지 헷갈릴 때
3. 제안서 톤을 더 세게 만들고 싶지만, 어디까지 써도 되는지 확인할 때
4. 심사 대응 차원에서 어떤 표현은 유지하고 어떤 표현은 피해야 하는지 알고 싶을 때
5. 공식 양식에 직접 옮기기 전에 최종 점검 포인트를 확인할 때

---

## 2. 최종 문안과의 관계

### 기준 문서
- `docs/papers/20_paper_a_scope/data_specification.md`

### 이 문서가 다루는 범위
- 문구 선택 이유
- 숫자 출처
- 구현 범위 vs 제안 범위 구분
- reviewer/동료가 수정할 가능성이 높은 항목
- 안전한 수정선 / 위험한 수정선

### 이 문서가 하지 않는 일
- 공식 양식 대신 최종 제출본 역할을 하지는 않음
- 새로운 주장이나 새로운 수치를 추가로 만들어내지 않음

## 3. 문서 작성 원칙

이번 `data_specification.md`는 아래 3개 원칙으로 정리했다.

### 3.1 사실 기반
- 코드/설정/증거 문서로 확인 가능한 내용만 단정적으로 작성
- 구현되지 않은 내용은 완료형으로 쓰지 않음

### 3.2 제안서 톤 유지
- 너무 기술문서처럼 움츠러들지 않도록
  - `지능형 진단 지원`
  - `운영 피드백 폐루프`
  - `단계적 고도화`
  - `제어 지원형 확장`
  같은 표현을 사용

### 3.3 심사 리스크 관리
- 다음 표현은 의도적으로 약화/제거함
  - RLHF 완료
  - fine-tuning 완료
  - 제어 파라미터 직접 자동 적용 완료
  - 사람 개입 없는 closed-loop 제어 구현 완료

이건 약하게 써서가 아니라, **근거 없이 강하게 쓰면 바로 공격받기 때문**이다.

---

## 4. 가장 중요한 스토리 라인

동료가 문서를 수정하더라도, 아래 큰 흐름은 유지하는 게 좋다.

### 현재 단계
- 문서 코퍼스 구축
- chunk_v3 기반 검색 인덱스 구축
- issue / sop 흐름 분리
- 운영 피드백 저장 및 export 경로 확보

### 제안서상 확장 단계
- 1차년도: RAG 기반 검색·진단 지원 체계 구축
- 2차년도: 조치 파라미터 추천 및 feedback 기반 고도화
- 3차년도: 현장 적용형 제어 지원/자율화 실증

즉, **지금은 검색·근거 제시 중심이지만, 제안서상 로드맵은 제어 지원형 AI 에이전트로 확장**되는 구조다.

---

## 5. 숫자 읽는 법

이 문서에서 숫자는 서로 다른 층위를 섞지 않도록 주의해서 썼다.

### 5.1 원본 수량
문서/파일/엔트리의 원본 규모를 뜻한다.

예:
- MyService 99,031 TXT
- GCB raw 16,354 entries
- SOP 507 files

### 5.2 retrieval corpus 수량
chunk_v3 파이프라인을 거쳐 검색용으로 적재된 chunk 수를 뜻한다.

예:
- myservice 329,206 chunks
- gcb 49,021 chunks
- sop 13,116 chunks

### 5.3 Paper A 실험 코퍼스 수량
논문용 실험 코퍼스는 전체 운영 코퍼스와 다르다.

예:
- Paper A 기준 578 parsed docs

### 실무 주의
동료가 수정을 하다가 이 세 숫자 층위를 섞으면 문서가 바로 어색해진다.
특히 아래 조합은 혼동하기 쉽다.

- `원본 99,031건` vs `활용 329,206건`
- `전체 chunk_v3 395,773건` vs `Paper A 578 docs`

## 6. 자주 수정될 수 있는 지점

### 6.1 데이터유형 칸
공식 양식에서는 너무 길게 쓰면 보기 안 좋다.

권장:
- `비정형 텍스트`
- `비정형 텍스트·JSON`
- `비정형 텍스트(PDF)`

비권장:
- 세부 구조를 데이터유형 칸에 길게 다 넣는 것

세부 구조 설명은 반드시 `세부항목` 칸으로 보낸다.

### 6.2 데이터 수집 내역의 `계`
여긴 서술형으로 써도 되지만, 너무 길게 쓰면 양식이 지저분해진다.

권장 방향:
- 연도별 `계`: 짧은 상태 설명
- 마지막 `계` 행: 전체 집계 한 줄 요약

예:
- `문서 코퍼스 395,773 chunks (원본 105,779건)`

### 6.3 AI 활용 계획의 마지막 행
가장 민감한 칸이다.

현재 문안은:
- 지금 구현 = 의사결정 지원
- 향후 = 제어 추천형 에이전트로 고도화

이 라인은 심사위원 관점에서 “품목개요서와 정합되는가”를 보는 포인트다.

## 7. 안전한 수정선 / 위험한 수정선

### 7.1 안전한 수정
아래는 동료가 손봐도 비교적 안전하다.

- 표현을 더 제안서답게 다듬기
- 표 셀 길이를 줄이기
- `지능형`, `고도화`, `확장`, `실증` 같은 제안서형 단어 강화
- `운영 피드백`, `근거 기반`, `의사결정 지원` 문구 정리
- 1/2/3차년도 로드맵 문구를 조금 더 분명히 쓰기

### 7.2 위험한 수정
아래는 근거 없으면 피하는 게 좋다.

- RLHF 완료로 쓰기
- retriever fine-tuning 완료로 쓰기
- direct control parameter 자동 적용 완료로 쓰기
- 사람이 개입하지 않는 closed-loop 제어 구현 완료로 쓰기
- 고정된 단일 모델 스택만 운영 중이라고 단정하기

## 8. 구현 기준으로 강하게 말해도 되는 것

아래는 비교적 자신 있게 말해도 된다.

### 8.1 구축 완료/운영 기반 확보
- 문서 코퍼스 구축
- chunk_v3 기반 retrieval corpus 구축
- doc_type-aware chunking 적용
- hybrid retrieval(BM25 + dense + RRF) 기반 구조 확보
- issue/sop 분기형 RAG 에이전트 구조 확보
- 상세 피드백 저장 및 export 체계 확보

### 8.2 고도화 기반 확보
- feedback export를 future fine-tuning data collection 기반으로 활용 가능
- 제어 추천형 에이전트로 확장 가능한 구조
- 운영형 피드백 루프 기반 확보
- 검색·진단·절차 안내를 하나의 산업 AI agent 흐름으로 통합할 수 있는 구조

## 10. 동료 검토 체크리스트

문서를 넘길 때 아래만 같이 보면 된다.

### 형식 체크
- [ ] 공식 양식 컬럼명과 맞는가
- [ ] 데이터유형 칸이 너무 길지 않은가
- [ ] `계` 컬럼이 비어 있지 않은가

### 내용 체크
- [ ] raw 수량과 chunk 수량을 혼동하지 않았는가
- [ ] Paper A 실험 코퍼스와 운영 코퍼스를 구분했는가
- [ ] 구현 범위와 향후 계획을 구분했는가

### 심사 대응 체크
- [ ] 제어 에이전트 방향성과 충돌하지 않는가
- [ ] 하지만 미구현 내용을 완료형으로 쓰지는 않았는가
- [ ] 1차년도/2차년도/3차년도 로드맵이 자연스러운가

## 12. 수정 제안이 들어오면 이렇게 처리

### 케이스 A: “더 있어 보이게 써야 한다”
- 먼저 이 문서의 7장(안전한 수정선) 안에서 조정
- `고도화`, `실증`, `지능형`, `확장 가능` 표현 강화

### 케이스 B: “품목개요서상 제어 에이전트여야 한다”
- 구현 완료형으로 바꾸지 말고
- `1차년도 구축 → 2차년도 추천 고도화 → 3차년도 현장 적용 실증`
  로드맵 문구를 강화

### 케이스 C: “숫자가 이상한데?”
- raw/source/chunk/eval 중 어느 층위 숫자인지 먼저 확인
- 출처 문서 재확인 후 수정
