# 문서 순차 판정 기반 답변 전략

## 목적

현재 agent는 retrieve + rerank 이후 상위 문서들을 한 번에 LLM에게 전달해 답변을 생성한다.  
이 방식은 다음 문제를 만든다.

1. 관련 문서와 비관련 문서가 같이 들어가 token 사용량이 커진다.
2. 답변이 절차 중심이 아니라 문서 전체 요약처럼 흐르기 쉽다.
3. SOP 절차 질문에서도 `Work Procedure` / `Workflow` 대신 준비/안전/개요가 앞에 길게 나온다.
4. 근거가 여러 문서에 섞이면 답변이 희석되고 상세도가 떨어진다.

이 문서는 위 문제를 줄이기 위한 **문서 순차 판정 기반 답변 전략**을 정리한다.

---

## 핵심 아이디어

상위 문서들을 한 번에 answer model에 넣지 않는다.  
대신 rerank 결과를 1번부터 차례대로 보면서, 각 문서 또는 챕터 그룹이 **질문에 직접 답하는 근거인지** 먼저 판정한다.

그리고:
- 관련 없는 문서는 버리고
- 관련 있는 문서/챕터만 누적하고
- 충분한 근거가 모이면 중단한 뒤
- **선별된 근거만** 최종 answer model에 넣는다.

즉, 구조는 아래와 같다.

1. retrieve + rerank
2. 문서/챕터별 relevance 판정
3. 관련 근거 누적
4. 충분하면 early stop
5. 최종 답변은 한 번만 생성

---

## 현재 방식과 제안 방식 비교

### 현재 방식

1. retrieve
2. rerank
3. 상위 문서 다수(예: top 10~20) 전체를 answer LLM에 전달
4. 답변 생성

문제:
- token 과다
- 관련 없는 문서 혼입
- 절차 질문에서 절차보다 general summary가 나옴
- 답변 길이에 비해 실제 근거 집중도가 낮음

### 제안 방식

1. retrieve
2. rerank
3. 상위 문서를 순차 판정
4. 관련 근거만 누적
5. 충분하면 중단
6. 최종 answer LLM에는 선별된 근거만 전달

효과:
- token 절감
- relevance 높은 근거만 사용
- 절차 질문에서 실제 작업 단계 중심 답변 가능
- 잘못 섞인 문서 때문에 답변이 흐려지는 문제 감소

---

## 중요한 설계 원칙

### 1. 문서마다 바로 답변 생성하지 않는다

각 문서를 읽을 때마다 부분 답변을 생성하는 방식은 비효율적이다.

문제:
- LLM 호출 횟수 증가
- 문서 간 답변 형태가 흔들릴 수 있음
- 잘못된 첫 문서를 기준으로 답변 방향이 굳을 수 있음

따라서 각 문서/그룹에 대해서는 **짧은 relevance 판정 + 근거 추출**만 수행하고,  
최종 답변 생성은 한 번만 수행한다.

### 2. 판정 단위는 가능하면 chunk가 아니라 그룹이어야 한다

권장 단위:
- `doc_id + section_chapter`

이유:
- SOP는 보통 하나의 절차가 여러 페이지에 걸쳐 있다.
- chunk 단위로 보면 같은 절차가 잘려 있고, 표지/목차/경고 페이지가 섞인다.
- 챕터 그룹 단위로 보면 실제 `Work Procedure` / `Workflow` 구간을 더 자연스럽게 볼 수 있다.

fallback:
- `section_chapter` 메타가 없는 경우 `doc_id + page window`
- 더 나아가 불가능하면 문서 단위

## 제안 아키텍처

### Step 1. Retrieve + Rerank

기존처럼 retrieve와 rerank는 유지한다.

입력:
- 질문
- task mode (`절차검색` / `이슈검색`)
- 장비/문서 필터

출력:
- relevance 기준으로 정렬된 candidate 목록

권장 형태:
- chunk 결과를 바로 쓰지 않고
- `doc_id + section_chapter` 그룹으로 재구성한 뒤 group rank를 만든다

예:
- Group 1: `global_sop_xxx / 10. Work Procedure / p9~14`
- Group 2: `global_sop_xxx / 8. Preparation / p6~8`
- Group 3: `global_sop_yyy / 4. Workflow / p11~16`

### Step 2. Sequential Evidence Judge

rerank 순서대로 각 group을 짧게 판정한다.

입력:
- 질문
- 현재 group의 title / section / snippet / pages

출력(JSON):

```json
{
  "relevant": true,
  "confidence": 0.93,
  "enough_to_answer": false,
  "reason": "질문의 교체 절차와 직접 관련된 Work Procedure 섹션",
  "key_sections": ["Work Procedure"],
  "key_pages": [9, 10, 11, 12],
  "evidence_type": "procedure"
}
```

필수 필드:
- `relevant`
- `confidence`
- `enough_to_answer`
- `key_pages`
- `reason`

### Step 3. Evidence Accumulation

판정 결과에 따라 누적한다.

- `relevant=false` → 버림, 다음 group으로 이동
- `relevant=true` → 근거 목록에 추가
- `enough_to_answer=true` → stop 가능

이때 중복 그룹은 제거한다.

중복 기준:
- `(doc_id, section_chapter)`

### Step 4. Early Stop

충분한 근거가 모였으면 더 이상 다음 문서를 보지 않는다.

stop 예시:
- 절차검색:
  - 첫 relevant group이 `Work Procedure` 또는 `Workflow`
  - confidence가 높음
  - 질문에 직접 대응되는 절차 단계가 있음
- 이슈검색:
  - 관련 사례 2~3개 확보
  - 증상/원인/조치 패턴이 충분히 모임

### Step 5. Final Answer Generation

최종 answer LLM에는 **누적된 selected evidence만** 전달한다.

즉, 기존의
- top 10~20 docs 전체
가 아니라
- selected groups 1~3개
- 핵심 page range
- 핵심 section
만 전달한다.

이렇게 하면:
- token 절감
- 답변 집중도 증가
- 절차 상세도 향상

---

## 절차검색과 이슈검색의 차이

### 절차검색

목표:
- 질문에 직접 대응하는 SOP 절차를 빠르게 찾고
- 실제 작업 순서를 자세히 설명

권장 정책:
- early stop 강하게
- `Work Procedure` / `Workflow` group 우선
- 관련 문서가 1개면 그 문서 중심으로 답변
- answer 입력 evidence는 1~2개 group이면 충분

stop 예시:
- `procedure` 유형 group 발견
- confidence ≥ 0.85
- `enough_to_answer=true`

### 이슈검색

목표:
- 증상과 유사한 사례를 모아
- 공통 원인/조치 패턴을 설명

권장 정책:
- early stop를 약하게
- 관련 사례 2~3개는 누적 허용
- answer 입력 evidence도 multi-doc 허용

stop 예시:
- relevant case 2~3개 확보
- 비슷한 원인/조치 패턴 확보

## 왜 이 방식이 현재 문제에 잘 맞는가

### 1. token 절감

지금은 top-N 문서를 한 번에 넣으므로 context가 커진다.  
제안 방식은 selected evidence만 answer에 넣으므로 최종 token이 줄어든다.

### 2. 답변 상세도 증가

절차검색에서 관련 없는 문서가 빠지면, answer model은 실제 procedure pages에 더 많은 token을 쓸 수 있다.

### 3. 절차 중심 답변 강화

판정 단계에서 `Work Procedure` / `Workflow`를 찾는 구조이므로,  
최종 답변도 자연스럽게 절차 중심으로 바뀐다.

### 4. chapter/group 구조와 잘 맞음

현재 chapter-aware/group expansion 구조는 유지하고,  
그 결과를 answer 직전 selection 단계에서 활용하면 된다.

즉:
- retrieve 자체를 바꾸는 게 아니라
- retrieve 이후 evidence selection을 정교화하는 방식이다.

---

## 우려 사항과 대응

### 우려 1. 문서별 판정 LLM 호출이 늘어나서 더 느려질 수 있음

대응:
- answer model과 별도 small judge prompt 사용
- 최대 판정 수 제한
  - 절차검색: top 3~5 groups
  - 이슈검색: top 5~8 groups
- early stop 적극 사용

### 우려 2. 첫 문서를 잘못 판정하면 답이 틀릴 수 있음

대응:
- `confidence`
- `enough_to_answer`
- `evidence_type`
를 구조화해서 받음
- confidence가 낮으면 다음 group 계속 확인

### 우려 3. 여러 문서를 합쳐야 하는 질문이 있음

대응:
- 절차검색은 single-doc 우선
- 이슈검색은 multi-doc 허용
- stop rule을 mode별로 다르게 설정

### 우려 4. section metadata가 없는 경우

대응:
- `section_chapter`가 있으면 group 단위
- 없으면 doc/page window fallback

## 구현 제안

### 새 노드 제안

`answer_node` 앞에 새 노드 추가:
- `evidence_select_node`

흐름:

1. retrieve
2. rerank
3. group normalize
4. `evidence_select_node`
5. selected evidence
6. answer
7. judge

### 내부 상태 제안

state에 추가:
- `candidate_groups`
- `selected_evidence_groups`
- `evidence_select_trace`
- `evidence_stop_reason`

### selection output schema

```json
{
  "relevant": true,
  "confidence": 0.91,
  "enough_to_answer": true,
  "reason": "질문의 교체 절차에 직접 대응하는 Work Procedure 섹션",
  "key_pages": [6, 7, 8, 9],
  "evidence_type": "procedure"
}
```

---

## 우선 적용 범위

### 1차
- `절차검색`에만 우선 적용
- group 단위 selection + early stop
- 최종 answer에는 selected evidence만 전달

### 2차
- `이슈검색`에도 적용
- multi-case accumulation 허용

### 3차
- selection trace를 UI/로그에 노출
- 왜 이 문서를 선택했고 왜 다음 문서를 건너뛰었는지 디버깅 가능하게 함

---

## 수용 기준

다음이 만족되면 성공으로 본다.

1. 절차검색에서 final answer 입력 evidence 수가 기존보다 줄어든다.
2. 절차검색 답변에서 `작업 절차` 섹션의 비중이 증가한다.
3. token 사용량이 감소한다.
4. 답변 초반 200자 안에 실제 작업 단계가 더 자주 등장한다.
5. 관련 없는 문서가 final evidence에 포함되는 비율이 감소한다.

---

## 결론

현재 문제는 retrieve 이후 상위 문서를 한 번에 answer model에 넣는 구조에서 온다.  
따라서 답변 품질을 개선하려면 retrieve 자체보다 **answer 직전 evidence selection 단계**를 추가하는 것이 효과적이다.

권장 방향은 다음과 같다.

- rerank 결과를 순차적으로 판정
- 관련 없는 문서는 버림
- 관련 있는 근거만 누적
- 충분하면 early stop
- 최종 답변은 선별된 근거만 사용

특히 SOP 절차검색에서는 이 방식이 token 절감과 절차 상세도 개선에 가장 잘 맞는다.
