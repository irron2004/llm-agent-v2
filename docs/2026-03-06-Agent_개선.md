# 2026-03-06 Agent 개선 정리

## 목적
- 사용자 입력 단계에서 혼란을 줄이고, 모델 변경(GLM/vLLM) 시 포맷 불이행으로 인한 오류를 방지한다.
- 업무 유형별 문서 검색 범위를 명확히 고정해 검색 품질과 일관성을 높인다.

## 개선 항목

### 1) `건너뛰기` 추천 표시 제거
- 대상: 첫 질문의 `auto_parse_confirm` 단계
- 변경: 파싱 실패/성공 여부와 무관하게 `건너뛰기`는 `recommended=false`로 고정
- 기대 효과: 사용자가 `건너뛰기`를 시스템 권장값으로 오해하지 않음

### 2) 모델 포맷 불이행 대응 (GLM 등)
- 문제: 모델이 지시문 대신 분석 텍스트(`Analyze the Request...`)를 출력해 파이프라인 오류 유발
- 변경:
  - `judge` 응답은 JSON 강제(`response_format=json_object`) 적용
  - `judge` 파싱 실패 시 재시도 폭주를 막는 안전 종료/완화 처리
  - `translate`, `query_rewrite` 결과 정규화(분석문/번호형 설명 제거)
- 기대 효과: 모델 변경 시에도 포맷 안정성 확보

### 3) 답변 완료 후 장비선택 버튼 오류 수정
- 문제: 답변 후에도 장비선택 버튼이 노출되어 클릭 시 에러 발생
- 변경:
  - 프론트: pending interrupt가 있을 때만 버튼 노출
  - 백엔드: pending 없는 resume 요청은 명확한 오류 코드/메시지 반환
- 기대 효과: 완료 상태에서 잘못된 resume 경로 차단

### 4) UI 용어 변경
- 변경: `SOP 조회` 표시명을 `절차조회`로 변경
- 주의: 내부 value(`sop`)는 호환 유지

### 5) 업무 선택 기반 고정 문서군 검색 규칙
- `절차조회` 선택 시: `SOP + TS + SETUP` 전체 검색
- `이슈조회` 선택 시: `gcb + myservice` 전체 검색
- 변경:
  - 세부 문서유형 추가 질문 단계는 만들지 않음
  - `task_mode` 선택값에 따라 백엔드에서 고정 doc_type 세트를 적용
- 기대 효과: 입력 단계는 단순하게 유지하면서 업무 의도와 검색 범위를 일치시킴

### 6) 단계형 탭 UI 도입 (질문 수 가시화)
- 요구 배경: 사용자가 "몇 개를 더 답해야 하는지" 즉시 파악할 수 있어야 함
- 변경:
  - 상단 탭/스텝 표시: 예) `언어 | 장비 | 설비ID | 업무`
  - 진행 상태 표시: `총 N개 중 X번째`
  - 현재 단계 선택 완료 시 다음 단계로 자동 이동
  - 이전 단계는 탭 클릭으로 수정 가능
  - 마지막 단계에서만 `확인/진행` 활성화
- 기대 효과: 입력 피로도 감소, 완료 예측 가능성 증가

## 수용 기준 (Acceptance Criteria)
- `건너뛰기`는 어떤 조건에서도 추천 표시가 되지 않는다.
- GLM 계열 모델에서도 `judge` 파싱 오류로 재시도 루프가 과도하게 반복되지 않는다.
- 답변 완료 후 장비선택 버튼 오류가 재현되지 않는다.
- UI에 `절차조회`가 표시되고 기존 내부 로직은 깨지지 않는다.
- `절차조회`/`이슈조회` 선택 시 각각 지정된 문서군 전체가 자동 적용된다.
- 탭형 단계 UI에서 사용자는 전체 질문 수와 현재 진행 위치를 항상 확인할 수 있다.

## 구현 우선순위 제안
1. UI/매핑 변경: 1, 4, 5, 6
2. 안정화: 2, 3
3. 통합 테스트: interrupt/resume, doc_type 필터, 모델 포맷 불이행 시나리오
