# 2026-03-14 Agent 개발 운영 가이드

## 0) 신규 agent 5분 온보딩 (필독)
신규 agent가 이 문서의 핵심을 놓치지 않도록, 작업 전에 아래 6가지를 먼저 고정한다.

1. `one agent per worktree`를 먼저 확보한다.
2. [`docs/contracts/product-contract.md`](./contracts/product-contract.md)에서 관련 contract ID를 찾는다.
3. 작업이 비사소하면 [`docs/tasks/TASK_TEMPLATE.md`](./tasks/TASK_TEMPLATE.md)로 task 문서를 먼저 만든다.
4. 코드 수정 전에 task 문서에 `Allowed Files`와 `Verification Plan`을 먼저 채운다.
5. 작업 중 범위가 늘어나면 코드를 먼저 바꾸지 말고 task 문서를 먼저 갱신한다.
6. 종료 시 검증 명령 실행 결과(성공/실패, 이유)를 task 문서에 남긴다.

작업 시작 체크리스트(복붙용):

```md
- [ ] 현재 worktree를 다른 agent가 사용 중이 아님
- [ ] 관련 contract ID를 `product-contract.md`에서 식별함
- [ ] (비사소 작업인 경우) task 문서를 생성/갱신함
- [ ] `Allowed Files`를 먼저 확정함
- [ ] `Verification Plan` 명령을 먼저 확정함
```

- 런타임별 래퍼:
  - OMC: [`task-start-kickoff`](../.omc/skills/task-start-kickoff/SKILL.md)

STOP 조건(필수):

- worktree 분리를 확신할 수 없으면, 기존 디렉토리에서 수정하지 말고 새 worktree를 만든다.
- 관련 contract ID를 식별하지 못했으면, 코드 수정 전에 task 문서에 `Contracts To Preserve: None (이유)`를 먼저 남기고 상태를 `blocked`로 표시한다.
- `Verification Plan`을 아직 정하지 못했으면 코드 수정을 시작하지 않는다.

## 1) 목적
- 여러 agent가 병렬로 작업할 때 발생하는 기능 rollback, 기존 동작 누락, 작업 범위 오염을 줄이기 위한 운영 규칙을 정의한다.
- 요구사항을 단순 메모가 아니라 보호 대상인 `contract`로 관리하고, 모든 비사소한 작업이 이를 참조하도록 표준화한다.
- 코드 변경 전후에 무엇을 보존해야 하는지, 어디까지 수정해도 되는지, 무엇으로 검증해야 하는지를 명시적으로 남긴다.

## 2) 문제 정의
여러 agent가 같은 repo를 만지면 다음 문제가 반복되기 쉽다.

- 작업 중인 내용이 다른 agent의 변경으로 덮어써진다.
- 사용자가 직접 개발해 둔 기능이 의도치 않게 rollback 된다.
- 새로운 기능 구현 과정에서 기존 API/UI 동작이 조용히 바뀐다.
- 변경 범위가 커지면서 unrelated file까지 같이 수정된다.
- 나중에 왜 특정 동작이 깨졌는지 추적하기 어려워진다.

문제의 핵심은 다음 두 가지다.

1. 여러 agent가 같은 worktree를 공유한다.
2. 보호해야 할 기존 동작이 문서와 테스트로 고정돼 있지 않다.

## 3) 핵심 원칙
1. 같은 worktree를 여러 agent가 공유하지 않는다.
2. 중요한 기존 동작은 `요구사항`이 아니라 `product contract`로 관리한다.
3. 비사소한 작업은 반드시 task 문서를 만든 뒤 시작한다.
4. contract를 바꾸는 작업은 문서와 테스트를 같은 변경에서 같이 수정한다.
5. prose만 믿지 말고 회귀 테스트로 보호한다.

## 4) 이 repo에 반영한 운영 장치

### 4.1 Product Contract
- 파일: [`docs/contracts/product-contract.md`](./contracts/product-contract.md)
- 역할:
  - 절대 조용히 깨지면 안 되는 동작을 모은다.
  - 각 항목에 contract ID를 부여한다.
  - 각 contract를 검증하는 테스트 명령을 같이 적는다.
- 현재 반영한 보호 항목 예시:
  - agent response metadata contract
  - interrupt/resume semantics
  - `retrieval_only=true` 동작
  - chat UI의 pending regeneration device flow

### 4.2 Task 문서 체계
- 가이드: [`docs/tasks/README.md`](./tasks/README.md)
- 템플릿: [`docs/tasks/TASK_TEMPLATE.md`](./tasks/TASK_TEMPLATE.md)
- 역할:
  - 작업 목표, 수정 허용 파일, 보호할 contract ID, 검증 명령을 한 곳에 고정한다.
  - 작업 도중 범위가 커지면 먼저 task 문서를 업데이트하게 만든다.
  - 나중에 “왜 이 파일이 바뀌었는지”와 “무엇을 검증했는지”를 추적할 수 있게 한다.

### 4.3 AGENTS 규칙 강화
- 파일: [`AGENTS.md`](../AGENTS.md)
- 이번에 추가한 규칙:
  - substantial work 전 `product-contract.md` 읽기
  - multi-file 작업 전 task 문서 생성 또는 업데이트
  - task 문서에 contract ID, allowed files, verification commands 명시
  - contract 변경 시 문서와 테스트를 같은 작업에서 같이 수정
  - 같은 git worktree에서 여러 coding agent 동시 작업 금지
  - `git status`에 unrelated dirty file이 있으면 user-owned change로 취급

## 5) 권장 운영 방식

### 5.1 가장 중요한 규칙: one agent per worktree
문서보다 먼저 지켜야 할 규칙이다.

- 금지:
  - 여러 agent가 같은 checkout 디렉토리에서 동시에 수정
- 권장:
  - agent마다 별도 branch 또는 `git worktree`

예시:

```bash
git worktree add ../llm-agent-v2-task-agent-api -b task/agent-api-guardrails
git worktree add ../llm-agent-v2-task-chat-ui -b task/chat-ui-regression-fix
```

이 규칙을 지키지 않으면, contract 문서가 있어도 파일 단위 덮어쓰기와 context loss를 완전히 막을 수 없다.

### 5.2 작업 시작 절차
1. `git status --short`로 현재 dirty files 확인
2. 같은 worktree를 다른 agent가 쓰고 있지 않은지 확인
3. 관련 contract ID를 [`docs/contracts/product-contract.md`](./contracts/product-contract.md) 에서 식별
4. 작업이 비사소하면 [`docs/tasks/TASK_TEMPLATE.md`](./tasks/TASK_TEMPLATE.md) 로 task 문서 작성
5. task 문서에 다음을 먼저 적고 코드 수정 시작
   - 목표
   - 보호할 contract ID
   - 수정 허용 파일
   - 검증 명령

### 5.3 작업 중 절차
- allowed files 밖으로 수정 범위가 커지면 코드를 먼저 바꾸지 말고 task 문서를 먼저 업데이트한다.
- contract에 걸린 동작을 건드리는 경우:
  - 변경 이유를 task 문서의 `Contracts To Update`에 기록
  - linked test를 같이 수정
  - `product-contract.md`도 같이 수정
- unrelated dirty file은 건드리지 않는다.

### 5.4 작업 종료 절차
1. diff가 allowed files 안에 있는지 확인
2. 관련 contract ID를 다시 읽고 여전히 맞는지 확인
3. task 문서의 verification commands 실행
4. contract 변경이 있었다면 contract 문서와 테스트가 같이 바뀌었는지 확인
5. 남은 리스크와 follow-up을 task 문서에 기록

### 5.5 비사소 작업 판정 기준 (task 문서 필수)
아래 중 하나라도 해당하면 `비사소 작업`으로 간주하고 task 문서를 필수로 만든다.

- 2개 이상 파일을 수정한다.
- backend와 frontend를 함께 건드린다.
- API payload, interrupt/resume, chat UI 분기처럼 회귀 위험이 큰 동작을 건드린다.
- 기존 contract ID와 직접 연결된 동작을 수정한다.
- 예상 diff가 커서(예: 대규모 리팩터링/흐름 변경) 허용 범위 관리가 필요하다.

### 5.6 검증 증빙 기록 포맷
검증 명령은 "실행했다"로 끝내지 말고, task 문서에 최소 아래 형식으로 남긴다.

- 기록 위치: task 문서에 `## Verification Results` 섹션을 만들고(없으면 추가) 그 아래에 기록한다.
- 최소 필드: `command`, `result(pass|fail|skip)`, `note`.
- 실패 시 추가 필드: `evidence`에 실패 로그 핵심 10~30줄 또는 대표 에러 1~3개를 그대로 남긴다.

```md
## Verification Results
- command: `uv run pytest tests/api/test_agent_response_metadata_contract.py -v`
  - result: pass
  - note: metadata key contract 유지 확인
- command: `cd frontend && npm run test -- src/features/chat/__tests__/chat-page-device-panel.test.tsx`
  - result: pass
  - note: pending regeneration device flow 회귀 없음
```

### 5.7 중단/핸드오프 포맷
작업을 다른 agent로 넘기거나 세션이 바뀌는 경우, 아래 정보를 task 문서 또는 handoff 메모에 남긴다.

- 기록 위치: task 문서에 `## Handoff` 섹션을 만들고(없으면 추가) 그 아래에 남긴다. 별도 handoff 문서는 명시 요청이 있을 때만 만든다.

- 현재 상태: done / in-progress / blocked
- 마지막으로 통과한 검증 명령과 결과
- 아직 남은 TODO와 우선순위
- `Allowed Files` 변경 여부와 이유
- contract 변경 예정 항목(`Contracts To Update`) 여부

## 6) 문서만으로는 부족한 이유
요구사항을 한 문서에 정리하는 것은 맞는 방향이다. 다만 그것만으로는 부족하다.

- 문서만 있고 테스트가 없으면:
  - agent가 문서를 읽고도 실수로 깨뜨릴 수 있다.
  - 깨졌는지 자동으로 확인되지 않는다.
- 테스트만 있고 작업 범위 문서가 없으면:
  - unrelated refactor나 범위 확장이 쉽게 섞인다.
- worktree 분리가 없으면:
  - 문서와 테스트가 있어도 파일 충돌과 덮어쓰기는 계속 생긴다.

따라서 실제 방어 체계는 아래 조합으로 본다.

- worktree 분리
- product contract
- task 문서
- 회귀 테스트
- AGENTS 강제 규칙

## 7) 현재 contract에 반영한 보호 항목
현재는 이미 테스트로 확인 가능한 동작만 먼저 contract에 올렸다.

- `C-API-001`
  - `/api/agent/run`, `/api/agent/run/stream` metadata contract 유지
- `C-API-002`
  - interrupt/resume 시 `thread_id` 연속성과 shared checkpointer 경로 유지
- `C-API-003`
  - `retrieval_only=true` 시 answer generation 전 interrupt
- `C-UI-001`
  - 이미 답변 완료된 assistant message가 있으면 missing-device prompt 비노출
- `C-UI-002`
  - pending regeneration device flow의 입력 의미 유지

세부 내용과 검증 명령은 [`docs/contracts/product-contract.md`](./contracts/product-contract.md) 를 따른다.

## 8) 실제 사용 예시

### 8.1 새 작업 시작
예: agent routing을 수정하려는 경우

1. 별도 worktree 생성
2. task 문서 생성
3. 관련 contract ID를 넣음
4. allowed files를 제한
5. 검증 명령을 미리 적음
6. 코드 수정
7. 검증 후 마감

예시 task 문서 skeleton:

```md
# Task: agent routing guardrails

## Contracts To Preserve
- C-API-001
- C-API-002

## Allowed Files
- backend/api/routers/agent.py
- backend/llm_infrastructure/llm/langgraph_agent.py
- tests/api/...

## Verification Plan
- uv run pytest tests/api/test_agent_response_metadata_contract.py -v
- uv run pytest tests/api/test_agent_interrupt_resume_regression.py -v
```

### 8.2 contract를 바꾸는 작업
예: 응답 metadata 필드를 바꾸려는 경우

- 금지:
  - 코드만 바꾸고 끝내는 것
- 필수:
  - `product-contract.md` 수정
  - 관련 테스트 수정
  - task 문서 `Contracts To Update`에 이유 기록

## 9) 운영자가 확인할 최소 체크리스트
- agent마다 worktree가 분리되어 있다.
- 비사소한 작업마다 task 문서가 있다.
- task 문서에 contract ID와 allowed files가 적혀 있다.
- 중요한 기존 동작은 contract와 테스트로 고정돼 있다.
- 종료 전에 verification commands를 실행했다.
- verification 결과가 command/result/note 형태로 기록돼 있다.
- 핸드오프 시 다음 agent가 바로 이어서 작업 가능한 상태 정보가 남아 있다.

## 10) 한계와 후속 보강
현재 반영은 문서와 규칙 수준이다. 완전 자동 방지는 아니다.

남은 한계:
- 사용자가 규칙을 무시하면 문서만으로 강제되지 않는다.
- 아직 모든 중요 기능이 contract에 올라가 있지는 않다.
- pre-commit/CI에서 task 문서나 contract를 강제 검사하지 않는다.

추천 후속 작업:
1. 중요한 UI/API 흐름을 더 많은 contract ID로 추가
2. PR 또는 commit 전 contract 관련 테스트를 자동 실행
3. task 문서 없이 큰 diff가 나면 경고하는 스크립트 추가
4. worktree 사용법을 팀 공통 운영 절차로 고정

## 12) 결론
여러 agent를 안정적으로 운영하려면 `요구사항 문서` 하나로는 부족하다. 이 repo에서는 요구사항을 `contract`로 올리고, 모든 큰 작업이 `task 문서 + verification`을 거치며, 동시에 `one agent per worktree`를 지키는 방식으로 운영한다.

이 문서를 기준으로 앞으로의 agent 작업은 다음 순서를 기본값으로 한다.

1. worktree 분리
2. contract 확인
3. task 문서 작성
4. 코드 수정
5. 검증 실행
6. contract 변경 시 문서와 테스트 동시 갱신
