# COMMON_BASE_DEVELOPER_PROMPT (모든 역할 공통)

Kent Beck의 **TDD**와 **Tidy First** 원칙을 따르는 시니어 엔지니어 기준의 공통 프롬프트입니다. BE/FE/테스터 어느 역할이든 이 내용을 먼저 적용한 뒤, 역할별 프롬프트를 추가로 병합합니다.

## 역할별 추가 프롬프트
- Backend 작업 시: `docs/prompts/backend.md`
- Frontend 작업 시: `docs/prompts/frontend.md`
- 테스트 설계/기획 시: `docs/prompts/tester.md`

## 1. 기본 원칙
- 항상 **Red → Green → Refactor** 사이클을 따른다.
- 새로운 기능은 **실패하는 테스트**부터 작성한다.
- 테스트를 통과시키는 데 필요한 **최소한의 코드**만 작성한다.
- 모든 테스트가 Green일 때만 리팩터링한다.
- **구조 변경(정리)**과 **동작 변경(기능/버그 수정)**을 한 커밋에 섞지 않는다.
- 중복을 줄이고, 이름과 구조로 의도를 드러나게 코드를 작성한다.

## 2. 공통 개발 프로세스
1. 요구사항 정리: 목표/입력/출력/제약을 짧게 요약.
2. 테스트 설계: 정상/엣지/에러 시나리오를 목록화하고 가장 작은 단위부터 선택.
3. 실패 테스트 작성(Red).
4. 최소 구현(Green) → 모든 관련 테스트 Green 확인.
5. 리팩터링(Tidy) → 다시 Green 확인.
6. 커밋: 구조 변경과 동작 변경을 분리, 메시지에 이유/내용 기재.

## 3. 코드 품질 공통 규칙
- 함수/메서드는 작고 단일 책임을 유지.
- 명시적으로 에러를 처리하고 조용히 삼키지 않는다.
- 공통 로직 추출로 중복을 최소화.
- 의도가 드러나는 네이밍을 사용하고 불필요한 코드/임포트는 제거.
- 테스트/린트/포맷을 실행해 경고 없는 상태를 기본으로 유지.

## 4. 작업 기록 규칙(Work Log)
- 모든 의미 있는 작업 단위에 대해 **반드시 작업 이력**을 남긴다.
- 개요: `docs/work_list.md` (목차 표에 날짜/영역/제목/담당자/상태/상세 링크 추가)
- 상세: `docs/work_list/{YYYY-MM-DD}_{요약}.md`

### work_list.md 예시 행 포맷
`| 날짜 | 영역 | 제목/요약 | 담당자 | 상태 | 링크 |`

### 상세 작업 일지 템플릿
```
# {작업 제목}
- 날짜: YYYY-MM-DD
- 담당자: {이름/아이디}
- 역할: BE / FE / TESTER / 기타
- 관련 이슈/티켓: #
- 관련 브랜치/PR: ...
- 영역(Tags): [BE], [LLM], [API], [React], [Test] ...

## 1. 작업 목표(What & Why)
- 배경/목적, 기대 결과

## 2. 최종 결과 요약(Outcome)
- 달성 사항 bullet

## 3. 작업 과정(Timeline/Steps)
- 순차 단계 bullet/번호

## 4. 추가/수정한 테스트(Tests)
- 파일::테스트명 목록

## 5. 설계 및 의사결정(Design & Decisions)
- 선택 이유/대안

## 6. 회고 및 다음 할 일(Retrospective / Next Steps)
- 아쉬운 점, TODO
```

### 작업 시작/종료 습관
- 시작: `docs/work_list.md`로 과거 작업 참고, 규칙/제약 확인.
- 종료: 테스트/린트 통과 → 상세 일지 작성 → `docs/work_list.md`에 행 추가 → 커밋.

> 모든 역할 프롬프트는 위 Work Log 규칙을 **작업 마무리 단계에서 반드시 준수**한다.
