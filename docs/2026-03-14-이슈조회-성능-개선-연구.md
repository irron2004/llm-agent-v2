# 이슈 조회 성능 개선 연구

날짜: 2026-03-14
대상: `task_mode=issue` (doc_type: myservice, gcb, ts)

## 1. 데이터 현황

### 1.1 chunk_v3 인덱스 doc_type 분포

| doc_type | 청크 수 | 비율 |
|----------|---------|------|
| myservice | 329,206 | 83.2% |
| gcb | 49,021 | 12.4% |
| ts | 760 | 0.2% |
| (참고: sop) | 13,116 | 3.3% |
| (참고: setup) | 3,583 | 0.9% |
| **합계** | **395,773** | |

이슈 조회 대상 3종(myservice+gcb+ts) = **379,987건 (96.0%)**

### 1.2 myservice 구조

각 유지보수 이력이 4개 섹션(status/action/cause/result)으로 **별도 청크**로 저장됨.

| 섹션 | 청크 수 | 중앙값(자) | 평균(자) | P10 | P90 |
|------|---------|-----------|---------|-----|-----|
| status | 78,953 | 46 | 64 | 29 | 107 |
| cause | 81,524 | **34** | 45 | 21 | 68 |
| action | 81,336 | 221 | 301 | 80 | 610 |
| result | 87,393 | 82 | 176 | 19 | 575 |

**핵심 특성:**
- 빈 콘텐츠 없음 (0%). 단, **cause 섹션 중앙값 34자**로 매우 짧음
- status도 중앙값 46자로 짧아, 이슈 증상을 충분히 기술하지 못하는 경우 다수
- action(221자)과 result(82자)가 상대적으로 정보량 많음
- 한 이슈의 정보가 4개 별도 청크에 분산 → 검색 시 한 청크만 hit되면 **나머지 3개 섹션 정보 유실**

### 1.3 gcb 구조

각 GCB 이슈가 summary/detail/timeline 등으로 **별도 청크**로 저장됨.

| 섹션 | 청크 수 | 중앙값(자) | 평균(자) |
|------|---------|-----------|---------|
| summary | 16,228 | 103 | 105 |
| detail | 16,485 | 1,256 | 1,297 |
| timeline | 15,259 | 1,375 | 1,397 |
| background | 257 | 488 | 728 |

고유 문서 수: **13,848건** (대부분 summary + detail 2청크 이상)

**핵심 특성:**
- 코드의 gcb_parser.py가 정의하는 question/resolution 구조와 **실제 chunk_v3 데이터의 summary/detail/timeline 구조가 다름**
- 실제 데이터는 question/resolution이 아닌 summary(103자 요약) + detail(1,256자 상세) + timeline(1,375자 이력) 형태
- summary만 검색되면 상세 정보 부족, detail만 검색되면 맥락 없음
- 영문 기반이나 nori analyzer 적용 중 → 영문 검색 정밀도 제한 가능

### 1.4 ts 구조

| 항목 | 값 |
|------|-----|
| 청크 수 | 760 |
| 형태 | VLM 파싱된 PDF → markdown |
| content 형식 | `# Trouble Shooting Guide [증상명]` + 테이블/도표 |
| device | 대부분 "ALL" |

**핵심 특성:**
- 760건 적재됨 (구조화된 alarm/symptom/cause/action 포함)
- 다른 doc_type 대비 **가장 구조화**된 이슈 정보
- 단, 전체 이슈 문서의 0.2%로 매우 적음

## 2. 현재 이슈 검색 파이프라인 분석

### 2.1 검색 흐름

```
사용자 쿼리
  → route_node: route='general' 하드코딩 (task_mode=issue일 때)
  → mq_node: general_mq 프롬프트로 EN+KO 검색어 생성
  → retrieve: myservice+gcb+ts 필터 + dense/BM25 hybrid → top 20
  → expand_related: section expansion
  → answer_node: issue_ans 프롬프트 → 사례 요약 목록
  → issue_confirm → issue_case_selection → issue_detail_answer
```

### 2.2 식별된 병목

#### B1. general_mq 사용 (높은 영향)

`task_mode=issue`에서 `route='general'`로 하드코딩되어 `general_mq` 프롬프트가 사용된다.
`ts_mq`에는 "root cause, check, reset, mitigation, sensor, valve" 등 트러블슈팅 특화 키워드 확장이 있지만, `general_mq`는 범용 변형만 생성한다.

- **영향**: 이슈 관련 검색어 품질 저하 → recall 감소
- **개선**: issue 전용 `issue_mq_v2.yaml` 프롬프트 추가

#### B2. 섹션 분리로 인한 정보 분산 (높은 영향)

myservice는 status/cause/action/result가, gcb는 summary/detail/timeline이 별도 청크.
검색에서 한 섹션만 hit되면 같은 이슈의 다른 섹션 정보가 답변에 포함되지 않는다.

- **영향**: 이슈 요약 시 "원인은 있는데 조치가 없음" 또는 "조치는 있는데 증상이 없음" 발생
- **개선**: expand_related에서 동일 doc_id의 다른 섹션 청크를 자동 병합

#### B3. MAX_ANSWER_REFS=5 vs 10건 사례 (중간 영향)

answer_node에서 REFS를 5개로 절삭 후 LLM에 전달하지만, UI에는 최대 10개 사례를 표시.
6~10번째 사례는 LLM이 요약 본문에서 인용하지 못한 채 선택지로만 노출된다.

- **영향**: 사용자가 6~10번 선택 시 요약 정보가 부정확할 수 있음
- **개선**: issue 모드 전용 `MAX_ISSUE_REFS=10` 상수 분리

#### B4. REFS에 doc_type/section_type 미설명 (중간 영향)

REFS 포맷에 `doc_type`과 `chapter`(section_type) 값은 포함되지만, LLM에게 각 값의 의미를 설명하지 않는다.
gcb의 summary/detail, myservice의 cause/action 등을 LLM이 구분하기 어렵다.

- **영향**: 답변에서 섹션별 정보를 적절히 배치하지 못함
- **개선**: issue_ans 프롬프트에 섹션 유형 설명 추가

#### B5. 상세 답변 추가 검색 부재 (중간 영향)

`issue_detail_answer_node`는 1단계 검색 결과에서 `selected_doc_id`와 일치하는 REFS만 필터링하여 재사용한다. 추가 ES 검색이 없으므로, 해당 문서의 모든 섹션이 1단계 결과에 포함되지 않았다면 상세 답변 컨텍스트가 부족하다.

- **영향**: 상세 답변 품질 저하
- **개선**: detail 단계에서 `selected_doc_id` 기반 doc-level fetch 추가

#### B6. gcb 영문 + nori analyzer 미스매치 (낮~중간 영향)

gcb 콘텐츠는 영문이나, ES 인덱스에 nori analyzer가 적용되어 있다.
nori는 한국어 형태소 분석기이므로 영문 검색 정밀도가 떨어질 수 있다.

- **영향**: gcb BM25 검색 품질 저하
- **개선**: `search_text.english` sub-field 추가

## 4. 주요 데이터 발견사항 정리

### Scientist 보고서 검증 결과

| 주장 | 실제 | 판정 |
|------|------|------|
| myservice 62.2%가 빈 콘텐츠 | 빈 콘텐츠 **0%** (전수 확인) | ❌ **오류** |
| ts chunk_v3에 0건 적재 | **760건** 적재됨 | ❌ **오류** |
| gcb가 question/resolution 구조 | 실제는 **summary/detail/timeline** 구조 | ❌ **오류** |
| cause 섹션 중앙값 33자 | 중앙값 **34자** (거의 정확) | ⚠️ 근사 |

Scientist가 로컬 data 디렉토리의 원본 파일을 기준으로 분석하거나, 코드 파서 구조만 보고 추정한 것으로 보임.
실제 chunk_v3 인덱스에 적재된 데이터와는 상당한 차이가 있음.

### 실제로 중요한 데이터 특성

1. **myservice cause 섹션이 34자로 매우 짧음** → 원인 분석 정보가 구조적으로 부족
2. **섹션 분리 저장** → 한 이슈의 정보가 4개(myservice) 또는 3개(gcb) 청크에 분산
3. **gcb 실제 구조가 parser 코드와 다름** → gcb_parser.py의 question/resolution 파싱이 현재 데이터에 맞지 않을 수 있음
4. **ts가 760건으로 소량이지만 가장 구조화됨** → 이슈 검색 정확도에 기여 가능
