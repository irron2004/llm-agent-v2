# 2026-03-05 UI 개선 계획 (Auto-Parse 확인형 UX + Task 선택형 검색)

## 1. 배경
- 현재 흐름은 사용자가 질문하면 바로 `auto_parse -> 검색 -> 답변`으로 진행된다.
- 이 과정에서 사용자 의도와 다른 파싱 결과(언어/디바이스)가 그대로 검색에 반영될 수 있다.
- 특히 이번 주 이슈인 `SOP가 GCB에 밀려 검색되지 않는 문제`를 해결하려면, 검색 전 사용자가 Task(검색 범위)를 명시적으로 선택해야 한다.

## 2. 목표
- 검색 전 사용자 확인 단계를 추가한다.
- 사용자가 숫자로 빠르게 선택할 수 있는 UX를 제공한다.
- Task 선택(`SOP 조회` / `이슈조회` / `전체조회`)에 따라 검색 대상 문서군을 제어한다.
- `SOP 조회` 선택 시 SOP 문서가 GCB에 밀리지 않도록 retrieval scope를 고정한다.

## 3. 요구사항 정리
### 3.1 필수 UX
- 질문 입력 후, 답변 전에 순차 질문 UI를 보여준다.
- 질문 순서: `언어 -> 디바이스 -> 장비ID(equipment_id) -> task`.
- 각 단계는 번호 선택 가능해야 한다.
- auto_parse 추출값이 있으면 1번 옵션에 `(Recommend)` 표시한다.

### 3.2 언어 선택 예시
- `너가 대답하기 원하는 언어는 무엇이니?`
- `1. {auto_parse 언어} (Recommend)`
- `2. 영어`
- `3. 중국어`
- `4. 일본어`

### 3.3 디바이스 선택 예시
- `검색할 장비는 무엇이니?`
- `1. {auto_parse device} (Recommend)`
- `2. {후보 device 1}`
- `3. {후보 device 2}`
- `...`

### 3.4 Task 선택 예시
- `검색 task를 선택해줘`
- `1. SOP 조회`
- `2. 이슈조회 (Recommend)`
- `3. 전체조회`
- 규칙:
- `SOP 조회` 선택 시 `doc_type = sop`만 검색
- `이슈조회` 선택 시 `doc_type in [myservice, gcb, ts]`만 검색
- `전체조회` 선택 시 기존 검색 스코프(필터 없음) 유지

## 4. 제안 UX 플로우
### 4.1 전체 시나리오
1. 사용자가 질문 입력
2. 백엔드가 `auto_parse`만 먼저 수행
3. 프론트가 확인 패널 표시 (숫자 입력 + 버튼 클릭 둘 다 지원)
4. 사용자 선택 완료
5. 선택값을 포함해 retrieval/answer 실행

### 4.2 입력 방식
- 방식 A: 버튼 클릭 (권장)
- 방식 B: 채팅 입력창에 숫자 입력 (`1`, `2`, `3` ...)
- 각 단계에서 `0` 입력 시 기본 추천값으로 진행 또는 건너뛰기

### 4.3 UI 배치
- 채팅 본문 하단 입력창 위에 `GuidedSelectionPanel` 렌더
- 현재 `pendingDeviceSelection` 패널과 동일 위치/패턴으로 구현
- 단계 진행 시 같은 패널 내부만 업데이트 (모달 신규 도입 없이 MVP 진행)

## 5. 상태 설계 (Frontend)
### 5.1 신규 상태 모델
```ts
type GuidedStep = "language" | "device" | "equip_id" | "task" | "done";
type TaskMode = "sop" | "issue";

interface GuidedOption {
  value: string;
  label: string;
  recommended?: boolean;
}

interface GuidedSelectionState {
  threadId: string;
  question: string;
  step: GuidedStep;
  languageOptions: GuidedOption[];
  deviceOptions: GuidedOption[];
  equipIdOptions: GuidedOption[];
  taskOptions: GuidedOption[];
  selectedLanguage?: string;
  selectedDevice?: string;
  selectedEquipId?: string;
  selectedTask?: TaskMode;
}
```

### 5.2 우선순위 규칙
- 최종 검색 파라미터 우선순위:
- `user selection > auto_parse result > system default`

## 6. 백엔드/계약 변경 계획
### 6.1 인터럽트 타입 확장
- 현재 interrupt type(`device_selection`, `retrieval_review`, `human_review`)에 아래 타입 추가:
- `auto_parse_confirm`

### 6.2 2단계 실행 계약
- 1차 호출: auto_parse 수행 + interrupt payload 반환
- 2차 호출(resume): 사용자 선택 포함해 retrieval/answer 수행

### 6.3 요청 필드(추가)
```json
{
  "target_language": "ko|en|zh|ja",
  "task_mode": "sop|issue",
  "filter_devices": ["..."],
  "filter_equip_ids": ["..."],
  "filter_doc_types": ["sop"] or ["myservice","gcb","ts"]
}
```

### 6.4 응답 메타(추가)
- `metadata.selected_language_source = user|auto_parse|default`
- `metadata.selected_task_mode = sop|issue`
- `metadata.applied_doc_type_scope = [...]`

## 7. 검색 스코프 정책
### 7.1 Task별 강제 스코프
- `task_mode=sop`: `filter_doc_types=["sop"]`
- `task_mode=issue`: `filter_doc_types=["myservice","gcb","ts"]`
- `task_mode=all`: `filter_doc_types=None` (전체 검색, 기존 동작)

### 7.2 충돌 처리
- 사용자가 task를 선택한 경우, 기존 라우터/노드 분류 결과보다 task 스코프를 우선 적용한다.
- 로그에 `scope_by_task_override=true`를 남긴다.

## 11. 수용 기준 (Acceptance Criteria)
- 사용자는 답변 전 4개 선택 단계(언어/디바이스/장비ID/task)를 볼 수 있다.
- 각 단계는 추천 옵션이 있으면 `(Recommend)`가 노출된다.
- 숫자 입력만으로 전체 선택 과정을 완료할 수 있다.
- `task=sop`일 때 SOP 외 문서 타입이 검색 결과에 포함되지 않는다.
- `task=issue`일 때 `myservice/gcb/ts` 외 문서 타입이 검색 결과에 포함되지 않는다.
- `task=all`일 때 기존 검색 동작처럼 전체 스코프에서 검색된다.

## 12. 리스크 및 대응
- 리스크: 단계 증가로 응답 체감 속도 저하
- 대응: 기본값 빠른 확정(`0=추천값 사용`) 제공
- 리스크: 옵션 과다 시 UX 복잡도 증가
- 대응: 상위 후보만 노출 + 직접입력 fallback 제공
- 리스크: 기존 node 기반 분류와 충돌
- 대응: task 선택 시 명시적 override 정책 적용
