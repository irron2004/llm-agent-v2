# 2026-03-09 Agent 분기 개선

## 상황

사용자가 guided selection에서 **"절차조회" / "이슈조회" / "전체조회"**를 선택하면, 이 `task_mode`는 **검색 필터링에만** 사용되고, 답변 생성 방식에는 전혀 반영되지 않는다.

### 현재 아키텍처

```
task_mode (사용자 선택)          route (LLM 분류)
    │                              │
    ▼                              ▼
검색 필터링만 사용              답변 템플릿 선택
- sop → SOP 문서만             - setup → setup_ans (단계별 절차)
- issue → gcb/myservice/ts     - ts → ts_ans (원인/진단/조치)
- all → 필터 없음              - general → general_ans (일반 답변)
```

**문제: task_mode와 route가 완전히 독립적**
- `task_mode="issue"`로 GCB 문서를 검색해도, route_node가 질문을 "general"로 분류하면 일반 답변 템플릿 사용
- GCB/이슈 문서에 특화된 답변 형식이 없어 **원문을 그대로 나열**

### 현재 그래프 플로우

```
Auto-parse 모드 (현재 사용):
  START → auto_parse → auto_parse_confirm → history_check
    → [query_rewrite] → translate → route(LLM) → [mq | prepare_retrieve]
    → retrieve → expand_related → answer → judge → [retry loop | done]

Legacy 모드 (미사용):
  START → route(LLM) → [mq | prepare_retrieve] → ...
```

## 문제

### 1. task_mode가 답변 방식에 반영 안 됨
- "이슈조회" 선택 → GCB 문서 검색 → 하지만 답변은 일반 RAG 형식
- 사용자가 원하는 것: **관련 GCB 사례만 필터하여 요약 표시**
- 실제 결과: GCB 원문을 [1], [2]... 로 그대로 복붙

### 2. 이슈 문서에 맞는 답변 프롬프트 없음
- `general_ans_v2.yaml`: "REFS 라인만 증거로 사용하세요" → 요약 지시 없음
- GCB 문서 특성: 이슈 내용 + 조치 사항 + 진행 상태 포함
- 필요한 형식: 관련 사례별 요약 (이슈 내용 / 조치 / 현재 상태)

### 3. route_node LLM 분류가 task_mode와 충돌
- `router_v2.yaml`은 질문 내용만으로 setup/ts/general 분류
- "웨이퍼 치핑 이유" 같은 질문은 "general"로 분류될 수 있음
- task_mode="issue"인데 route="general" → 이슈 특화 답변 불가
- **LLM 분류 자체가 불필요** — task_mode가 이미 사용자의 의도를 명시

### 4. 답변 언어 처리 방식 (결정사항)
- guided selection에서 사용자가 언어를 선택 (`target_language`)
- 답변 생성은 **언어별 answer prompt 선택 방식**으로 처리
  - ko: `*_ans_v2.yaml`
  - en: `*_ans_en_v2.yaml`
  - zh: `*_ans_zh_v2.yaml`
  - ja: `*_ans_ja_v2.yaml`
- 동적 언어 지시(프롬프트 문자열 append) 방식은 사용하지 않음
- **결정:** 언어별 v2 답변 프롬프트를 실제로 생성하고, 해당 언어 프롬프트만 선택해 사용
- 구현 범위:
  - 기존 route용: `setup_ans_en/zh/ja_v2.yaml`, `ts_ans_en/zh/ja_v2.yaml`, `general_ans_en/zh/ja_v2.yaml`
  - 신규 issue용: `issue_ans_v2.yaml` + `issue_ans_en/zh/ja_v2.yaml`
- `load_prompt_spec("v2")`에서 위 파일을 로딩하도록 확장

### 5. task_mode="issue" 검색 범위 (결정사항)
- 현재 코드: `expand_doc_type_selection(["myservice", "gcb", "ts"])` (L3344-3346)
- **결정:** 이슈조회 범위에 `ts`를 포함한다
- 즉, 이슈조회 문서 범위는 `gcb/myservice/ts`로 고정

### 6. 반복 출력 문제 (복합 원인)

**원인 1: 소스 데이터 반복**
- GCB 문서 확장(expand) 시 동일 내용이 중복 포함될 수 있음
- 청크 내에 동일 문장이 반복되는 파싱 품질 문제

**원인 2: LLM 생성 반복**
- ollama 경로: `repeat_penalty=1.1` 적용됨 (settings.py:288, ollama engine:86,146)
- vLLM 경로: 동일 제어 파라미터 전달 없음 (vllm engine:49) — vLLM 사용 시 반복 위험

**원인 3: judge 재시도 루프**
- judge 실패 시 retry_expand → retry → retry_mq 순으로 의도적 루프 (should_retry:2249)
- 재시도 과정에서 동일/유사 답변이 반복 생성될 수 있음
- 재시도 상한은 있으나 (`max_attempts`), 루프 중 사용자에게 중간 결과가 보일 수 있음

### 7. 후속 질문 시 필터 유지 정책 (Open Question)
- 현재 코드에서는 후속 질문(`needs_history=True`)이고 strict 필터가 켜져 있으면 이전 `selected_doc_types`를 유지함
- 의문: 사용자가 주제를 바꾼 질문을 했을 때도 이전 필터를 유지할지, 자동으로 해제할지
- 결정 필요:
  - 옵션 A: 유지 우선 (현재 방식)
  - 옵션 B: 새 주제 감지 시 필터 초기화
  - 옵션 C: 매 턴 사용자에게 유지/초기화 선택 제공

## 개선 방안

### 목표 아키텍처

```
task_mode (사용자 선택)  →  route 고정 + 답변 템플릿 + MQ 템플릿 + judge
    │
    ├─ sop   → route="setup"  → setup_mq → setup_ans → judge_setup
    ├─ issue → route="general" → general_mq → issue_ans(신규) → judge_issue(신규)
    └─ all   → route="general" → general_mq → general_ans → judge_general

* LLM 기반 route 분류 제거 (task_mode가 설정된 auto_parse 모드에서)
* 배경: `all` 모드에서 LLM route 분류 정확도가 낮아 오분류가 자주 발생함
* Legacy 모드 (task_mode 없음): 기존 LLM 분류 유지 (하위 호환)
```

### 1. route_node에서 LLM 분류 제거 → task_mode 기반 고정 라우팅

```python
def route_node(state, *, llm, spec):
    task_mode = state.get("task_mode")
    if task_mode:
        # task_mode 설정됨 → LLM 호출 없이 고정 라우팅
        route_map = {"sop": "setup", "issue": "general", "all": "general"}
        route = route_map.get(task_mode, "general")
    else:
        # Legacy fallback: LLM 기반 분류 유지
        ...기존 코드...
```

**장점:**
- LLM 호출 1회 절약 (응답 속도 개선)
- task_mode와 route 불일치 문제 완전 해소
- route_node가 task_mode를 모르는 문제 해결

**영향 범위:**
- `mq_node`: issue → route="general" → general_mq 사용 (범용 검색어 생성, doc type 필터가 문서 범위 제한)
- `judge_node`: issue → judge_issue_sys 사용 (신규 — 이슈/원인/조치/현황 4항목 구조 완결성 평가)
- `answer_node`: task_mode 기반 분기 추가 필요 (아래 참조)

- `PromptSpec` dataclass에 `issue_ans` 필드 추가 (+ `issue_ans_en/zh/ja`)
- `load_prompt_spec()`에서 `issue_ans` 로딩 추가

### 6. 이슈조회 → 다단계 대화형 플로우

사용자가 "이슈조회"를 선택하면 단일 응답이 아닌 **대화형 단계별 플로우**로 동작:

```
Step 1: 이슈 요약 목록 표시
  ┌─────────────────────────────────────────┐
  │ 질문과 관련된 이슈 N건을 찾았습니다:    │
  │                                         │
  │ [1] GCB-2026-001 (장비A)                │
  │     이슈: 웨이퍼 치핑 발생              │
  │     조치: 척 클리닝 수행                │
  │     현황: 해결                          │
  │                                         │
  │ [2] GCB-2026-015 (장비B)                │
  │     이슈: 동일 증상 반복 발생           │
  │     조치: 부품 교체                     │
  │     현황: 모니터링 중                   │
  │                                         │
  │ 번호를 선택하면 상세 내용을 확인합니다. │
  └─────────────────────────────────────────┘

Step 2: 사용자가 [1] 선택 → 해당 문서 상세 표시
  ┌─────────────────────────────────────────┐
  │ GCB-2026-001 상세                       │
  │                                         │
  │ ■ 이슈 내용: (핵심 원문 발췌 + 요약)     │
  │ ■ 원인 분석: ...                        │
  │ ■ 조치 내역: ...                        │
  │ ■ 관련 SOP: SOP-ABC-001 "척 클리닝 절차"│
  │                                         │
  │ → 관련 SOP 절차를 확인하시겠습니까?     │
  └─────────────────────────────────────────┘

Step 3: 사용자가 SOP 확인 요청 →
  Case A: 이슈 문서에 SOP 참조 있음
    → 해당 SOP 문서의 내용을 바로 표시

  Case B: 이슈 문서에 SOP 참조 없음
    → 이슈 내용 기반으로 관련 SOP 문서를 검색
    → 검색된 SOP 절차를 표시
```

**Phase 분리:**
- **Phase A (현재 개선 범위):** Step 1 — issue_ans 프롬프트로 요약 목록 생성
- **Phase B (후속 작업):** Step 2/3 — 상세 표시 + SOP 연계

**Phase A: Step 1 요약 목록**
- issue_ans 프롬프트로 요약 목록 생성
- 검색 결과 0건 시 대안 경로: "관련 이슈를 찾지 못했습니다. 전체조회로 전환하시겠습니까?" 안내 표시

**Phase B: Step 2/3 상세 + SOP 연계**

doc_id 매핑 구조:
- answer_node에서 `ref_json` 인덱스와 답변 내 번호의 매핑 테이블을 state에 저장
- 예: `issue_doc_map: {1: "doc_id_abc", 2: "doc_id_xyz"}`
- 사용자가 번호 선택 시 매핑 테이블에서 doc_id를 조회하여 원문 fetch

Step 2 상세 표시 정책:
- `PEMS`: 원문 전체 표시 허용 (이미지/도표 해석 문맥 보존 목적)
- `gcb/myservice/ts`: 원문 전체가 아니라 핵심 구간 발췌 + 구조화 요약 표시

Step 3 SOP 연계:
- 이슈 문서 내 SOP 참조 파싱 → 있으면 직접 fetch, 없으면 재검색 (task_mode를 "sop"로 전환하여 SOP 문서 검색)
- 검색 결과 0건 시: "관련 SOP를 찾지 못했습니다" 안내 + 플로우 종료

구현 메커니즘:
- LangGraph의 `interrupt` 메커니즘 활용 (auto_parse_confirm_node과 동일 패턴)
- Step 2/3 각각 새로운 interrupt 포인트 필요 (그래프에 노드 추가)
- 프론트엔드에서 번호 선택 UI 필요 (guided_selection_panel과 별도 컴포넌트)

## 구현 우선순위
