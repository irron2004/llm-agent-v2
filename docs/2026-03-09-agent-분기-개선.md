# 2026-03-09 Agent 분기 개선

## 상황

사용자가 guided selection에서 **"절차조회" / "이슈조회" / "전체조회"**를 선택하면, 이 `task_mode`는 **검색 필터링에만** 사용되고, 답변 생성 방식에는 전혀 반영되지 않는다.

### 현재 아키텍처

```
task_mode (사용자 선택)          route (LLM 분류)
    │                              │
    ▼                              ▼
검색 필터링만 사용              답변 템플릿 선택
- sop → SOP 문서만             - setup → setup_ans (단계별 절차)
- issue → gcb/myservice/ts     - ts → ts_ans (원인/진단/조치)
- all → 필터 없음              - general → general_ans (일반 답변)
```

**문제: task_mode와 route가 완전히 독립적**
- `task_mode="issue"`로 GCB 문서를 검색해도, route_node가 질문을 "general"로 분류하면 일반 답변 템플릿 사용
- GCB/이슈 문서에 특화된 답변 형식이 없어 **원문을 그대로 나열**

### 현재 코드 경로

| 구분 | 파일 | 위치 |
|------|------|------|
| task_mode 설정 | `langgraph_agent.py` | `auto_parse_confirm_node` (L3337-3350) |
| route 결정 | `langgraph_agent.py` | `route_node` (L1007-1021) |
| 답변 템플릿 선택 | `langgraph_agent.py` | `answer_node` (L2095-2173) |
| router 프롬프트 | `prompts/router_v2.yaml` | setup/ts/general 3분류 |
| 답변 프롬프트 | `prompts/general_ans_v2.yaml` | REFS 인용만 지시 |

## 문제

### 1. task_mode가 답변 방식에 반영 안 됨
- "이슈조회" 선택 → GCB 문서 검색 → 하지만 답변은 일반 RAG 형식
- 사용자가 원하는 것: **관련 GCB 사례만 필터하여 요약 표시**
- 실제 결과: GCB 원문을 [1], [2]... 로 그대로 복붙

### 2. 이슈 문서에 맞는 답변 프롬프트 없음
- `general_ans_v2.yaml`: "REFS 라인만 증거로 사용하세요" → 요약 지시 없음
- GCB 문서 특성: 이슈 내용 + 조치 사항 + 진행 상태 포함
- 필요한 형식: 관련 사례별 요약 (이슈 내용 / 조치 / 현재 상태)

### 3. route_node가 task_mode를 모름
- `router_v2.yaml`은 질문 내용만으로 setup/ts/general 분류
- "웨이퍼 치핑 이유" 같은 질문은 "general"로 분류될 수 있음
- task_mode="issue"인데 route="general" → 이슈 특화 답변 불가

### 4. task_mode="issue" 검색 범위에 ts 포함 여부
- 현재 코드: `expand_doc_type_selection(["myservice", "gcb", "ts"])` (L3344-3346)
- 이슈조회 목적에 ts(Troubleshooting) 문서가 포함되어야 하는지 확인 필요
- 사용자 의도가 "gcb/myservice만"이라면 ts 제거 필요

### 5. 반복 출력 문제 (복합 원인)

**원인 1: 소스 데이터 반복**
- GCB 문서 확장(expand) 시 동일 내용이 중복 포함될 수 있음
- 청크 내에 동일 문장이 반복되는 파싱 품질 문제

**원인 2: LLM 생성 반복**
- ollama 경로: `repeat_penalty=1.1` 적용됨 (settings.py:288, ollama engine:86,146)
- vLLM 경로: 동일 제어 파라미터 전달 없음 (vllm engine:49) — vLLM 사용 시 반복 위험

**원인 3: judge 재시도 루프**
- judge 실패 시 retry_expand → retry → retry_mq 순으로 의도적 루프 (should_retry:2249)
- 재시도 과정에서 동일/유사 답변이 반복 생성될 수 있음
- 재시도 상한은 있으나 (MAX_RETRIES), 루프 중 사용자에게 중간 결과가 보일 수 있음

## 개선 방안

### 1. task_mode를 답변 템플릿 선택에 반영

**방안 A (권장): task_mode 우선, route 보조**
```
task_mode="issue" → issue_ans 프롬프트 사용 (route 무시)
task_mode="sop"   → route에 따라 setup_ans / ts_ans 선택
task_mode="all"   → 기존대로 route 기반 선택
```

**방안 B: route에 task_mode 힌트 전달**
- router 프롬프트에 task_mode 정보 추가
- 단점: LLM이 힌트를 무시할 수 있음

### 2. 이슈 전용 답변 프롬프트 신규 작성 (`issue_ans_v2.yaml`)

구현 시 필요한 변경:
- `prompts/issue_ans_v2.yaml` 파일 생성
- `PromptSpec` dataclass에 `issue_ans` 필드 추가 (현재 없음)
- `load_prompt_spec()`에서 `issue_ans` 로딩 추가
- 언어별 변형 (`issue_ans_en`, `issue_ans_zh`, `issue_ans_ja`) 도 고려

```yaml
name: issue_ans
version: v2
description: Issue case summary answer for GCB/myservice queries (Korean).
system: |-
  당신은 반도체 설비 이슈 분석 RAG 어시스턴트입니다.
  - REFS에서 질문과 관련된 사례만 선별하여 답변하세요.
  - 관련 없는 REFS는 무시하세요.
  - 각 관련 사례를 다음 형식으로 요약하세요:
    ### [N] 문서ID (장비명)
    - **이슈**: 한 줄 요약
    - **원인**: 확인된 원인
    - **조치**: 수행한 조치 사항
    - **현황**: 현재 상태 (해결/진행 중/모니터링)
  - REFS가 비어있으면 관련 이슈를 찾지 못했다고 답변하세요.
  - 반드시 한국어로 답변하세요.
messages:
  - role: user
    content: |
      질문: {sys.query}
      REFS:
      {ref_text}
```

### 3. answer_node에서 task_mode 기반 분기 추가

```python
# answer_node 내부
task_mode = state.get("task_mode")
if task_mode == "issue":
    tmpl = spec.issue_ans  # 이슈 전용 프롬프트
elif task_mode == "sop":
    tmpl = templates.get(route, spec.setup_ans)  # route 기반, 기본값 setup
else:
    tmpl = templates.get(route, spec.general_ans)  # 기존대로
```

### 4. 반복 출력 방지 (다층 대응)

| 계층 | 대응 | 상태 |
|------|------|------|
| ollama 생성 | `repeat_penalty=1.1` 설정 | 적용됨 |
| vLLM 생성 | `frequency_penalty` / `repetition_penalty` 전달 | 미적용 — vLLM 사용 시 추가 필요 |
| REFS 전처리 | 중복 문장/청크 제거 로직 | 미구현 |
| judge 재시도 | 루프 중 사용자에게 중간 노출 방지 | 확인 필요 |

## 구현 우선순위

1. **이슈 전용 프롬프트 도입** — issue_ans_v2.yaml + PromptSpec 확장 + answer_node 분기
2. **REFS 중복 제거** — 전처리로 소스 데이터 품질 개선
3. **task_mode="issue" 검색 범위 확정** — ts 포함/제외 결정
4. **vLLM 경로 반복 방지** — vLLM 사용 시 frequency_penalty 전달
