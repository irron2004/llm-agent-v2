## 상황

RRF는 이전 문서와 계획에서 이미 중요한 개선 항목으로 정의되어 있었고, 일부 문서에서는 **완료**로 기록되어 있다.  
하지만 현재 실제 chat flow 런타임에서는 stage1 hybrid retrieval이 **app-level RRF**로 동작하지 않고, **ES native RRF 시도 후 `script_score` fallback**으로 동작하고 있다.

즉, 문제는 "RRF 계획이 없었다"가 아니라 다음에 가깝다.

- 계획: app-level RRF로 통일
- 문서: 완료로 기록
- 현재 런타임: stage1은 여전히 native ES RRF 경로 사용
- 결과: native RRF 실패 시 weighted hybrid fallback으로 검색 품질 흔들림

## 실제 런타임에서 확인된 증상

chat flow를 실제로 확인했을 때 RRF는 다음 에러로 실패했다.

```text
RRF search failed, falling back to script_score
[knn] unknown field [k]
```

이 의미는 명확하다.

1. 코드가 native ES RRF를 시도함
2. 현재 ES와 요청 형태가 맞지 않음
3. 검색은 결국 `script_score` 가중치 방식으로 처리됨

따라서 현재 실서비스 검색 방식은 기대했던 "RRF 기반 hybrid"가 아니라 사실상 다음이다.

```text
dense_weight * cosineSimilarity + sparse_weight * BM25
```

이 fallback은 동작은 하지만, 계획했던 RRF와 성질이 다르다.

## 왜 문제가 되나

### 1. bilingual chat flow에서 순위가 흔들림

현재 chat flow는 실제로 다음 흐름을 탄다.

- 한국어 질문 입력
- 영어 번역 생성
- `EN + KO` 검색어를 함께 사용
- hybrid retrieval 수행

관련 코드:

- [langgraph_agent.py:3837](/home/hskim/work/llm-agent-v2/backend/llm_infrastructure/llm/langgraph_agent.py:3837)
- [langgraph_rag_agent.py](/home/hskim/work/llm-agent-v2/backend/services/agents/langgraph_rag_agent.py)

이 구조에서 RRF는 EN/KO, dense/sparse 간 점수 스케일 차이를 덜 민감하게 만들어야 한다.  
그런데 지금은 weighted fallback이 동작하므로, 특정 broad 문서가 앞에 오고 정답 SOP가 뒤로 밀리는 현상이 발생한다.

### 2. dense-only 평가와 chat flow 평가가 달라짐

이전에 돌렸던 `question -> embed -> ES knn` 평가는 dense-only라서, 현재 문제를 정확히 설명하지 못한다.  
실제 문제는 chat flow에서 발생하는 **번역 + hybrid + fallback** 조합의 문제다.

### 3. 문서와 코드가 어긋나 있어 운영 판단이 틀어짐

문서만 보면 "RRF 완료"처럼 보인다.  
하지만 실제 운영은 RRF fallback 상태다.  
이 상태는 디버깅과 성능 평가 기준을 흐리게 만든다.

현재 RRF 관련 상태는 다음처럼 정리하는 것이 정확하다.

- 회귀(regression)

- 문서와 코드 동기화 실패

## 수정 목표

목표는 단순하다.

### 목표 1. stage1 hybrid retrieval을 실제 app-level RRF로 통일

즉, 다음 구조로 바꾼다.

```text
dense_search(query_vector)
  +
sparse_search(query_text)
  ->
rrf.py에서 app-level merge
```

### 목표 2. native ES RRF 의존 제거

현재 `_hybrid_search_rrf()`가 만드는 `sub_searches + rank.rrf` 경로를 제거하거나 비활성화한다.

### 목표 3. stage1/stage2의 병합 규칙을 동일하게 맞춤

이미 stage2 쪽은 app-level RRF 유틸을 쓰고 있으므로, stage1도 같은 규칙으로 통일한다.

### 목표 4. fallback이 숨어서 동작하지 않게 만듦

RRF 실패 후 자동 `script_score` fallback은 문제를 숨긴다.  
운영 기본값에서는 fallback에 의존하지 않도록 해야 한다.

## 수정 방안

## 1. `EsSearchEngine._hybrid_search_rrf()`를 app-level merge 방식으로 변경

수정 대상:

- [es_search.py](/home/hskim/work/llm-agent-v2/backend/llm_infrastructure/retrieval/engines/es_search.py)

현재:

- native ES RRF 요청 생성
- 실패 시 script_score fallback

변경 후:

1. `dense_search()` 실행
2. `sparse_search()` 실행
3. 각 결과를 `RetrievalResult` 또는 `EsSearchHit` 기준으로 정규화
4. `rrf.py` 유틸로 병합
5. 대표 hit 선정 + metadata 주입

핵심 포인트:

- stage1도 `rrf.py`를 사용하게 한다
- native ES RRF 요청을 더 이상 만들지 않는다
- `script_score`는 별도 explicit mode일 때만 사용하게 한다

## 2. RRF 메타데이터를 stage1에도 확실히 기록

문서상 기대한 메타:

- `rrf_dense_rank`
- `rrf_sparse_rank`
- `rrf_score`
- `rrf_k`

이 값이 실제 검색 결과 metadata에 들어가야 한다.  
그래야 왜 특정 문서가 올라왔는지 재현할 수 있다.

## 3. `script_score`는 fallback이 아니라 별도 모드로 격리

권장 방향:

- 기본: `use_rrf=True` -> app-level RRF
- 명시적 비교 실험에서만 `use_rrf=False` -> script_score

즉, `script_score`는 "조용한 장애 대응"이 아니라 "의도된 실험 모드"가 되어야 한다.

## 4. chat flow 기준 평가 스크립트 분리 또는 강화

현재 dense-only eval은 retrieval 엔진 비교용으로는 쓸 수 있지만, chat flow 품질을 설명하지 못한다.

필요한 것:

- `/api/agent/run` 또는 내부 그래프 경로를 통한 chat-flow retrieval 평가
- search query, top-k docs, fallback 여부, RRF metadata 저장

이 평가를 기준으로 전/후를 비교해야 한다.

## 5. page 확장 오류도 같이 수정

별도지만 영향이 크다.

현재 `expand_related` / page fetch 쪽에서 `NoneType` 관련 오류가 발생하고 있어, RRF만 고쳐도 page-hit이 완전히 회복되지 않을 수 있다.

따라서 RRF 수정과 함께 다음도 병행 점검한다.

- `fetch_doc_pages failed ... float() argument must be a string or a real number, not 'NoneType'`

이 항목은 RRF 자체는 아니지만, retrieval 품질 검증을 왜곡하는 요소다.

## 결론

현재 RRF 문제의 본질은 "RRF를 새로 설계해야 한다"가 아니다.

이미 있는 계획과 구현 일부를 **실제 stage1 런타임 경로에 다시 제대로 연결하는 작업**이다.

가장 중요한 판단은 다음이다.

- RRF는 문서상 완료였지만, 현재 운영 경로는 그 상태가 아니다.
- 따라서 우선 작업은 새로운 알고리즘 추가가 아니라 **경로 정합성 회복**이다.
- 이 정합성이 회복되기 전에는 dense-only 평가나 단순 ES 평가로는 실제 chat 품질을 설명할 수 없다.
