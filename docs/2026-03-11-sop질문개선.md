## 요구사항 정리

## 1. 확장 문서/참고 문서 페이지 표시 개선

### 문제

- 현재 페이지 이미지가 확대된 상태로 렌더링된다.
- 노트북 모니터 기준으로 페이지 상하/좌우가 잘려서 보인다.
- 우측 bar에서 특정 페이지 썸네일을 눌러도, 실제 preview는 클릭한 페이지와 다른 페이지로 열릴 수 있다.

### 목표

- 기본값이 `페이지 전체 보기(fit-to-page)`가 되도록 수정한다.
- 사용자는 스크롤 없이 페이지 전체 윤곽을 먼저 볼 수 있어야 한다.
- 사용자가 클릭한 페이지와 preview에 표시되는 페이지가 정확히 일치해야 한다.

### 구현 방향

- 기본 렌더링을 `contain` 기준으로 변경한다.
- 이미지/페이지 컨테이너의 높이를 화면 기준으로 제한한다.
- 확대는 기본값이 아니라 사용자 액션으로만 적용한다.
- 가능하면 `페이지에 맞춤` / `너비에 맞춤` / `원본 크기` 3가지 모드를 둔다.
- 우측 bar preview 데이터는 문서 단위 1장이 아니라, `expanded_page_urls` 기준의 페이지 단위 배열로 만들어야 한다.
- 클릭 이벤트는 `docIndex`가 아니라 실제 `previewIndex`로 매핑해야 한다.

### 제안

- 기본 모드는 `페이지에 맞춤`
- 사용자 확대/축소 버튼은 유지
- 동일한 확대 정책을 다음 두 화면에 같이 적용
  - 채팅 메시지 내부 참고 문서 미리보기
  - 우측 사이드바 `확장 문서/참고 문서`
- 페이지 클릭 버그는 확대 정책과 별도 이슈로 분리하지 말고, 같은 단계에서 같이 수정하는 것이 좋다.

### 현재 추정 원인

우측 bar의 `RetrievedDocsContent`는 preview용 배열을 문서 기준으로 1개씩 만들고 있다.

- `handleImageClick(index, pageIdx)`는 페이지 인덱스를 받지만
- 실제 `previewImages`는 `docs.map()` 기반이라 문서당 1개 아이템만 가진다
- 그래서 다른 페이지를 눌러도 첫 페이지 또는 문서 대표 페이지가 열릴 수 있다

관련 코드:

- [frontend/src/components/layout/index.tsx](/home/hskim/work/llm-agent-v2/frontend/src/components/layout/index.tsx)

반면 채팅 본문 쪽 `message-item.tsx`는 이미 페이지 단위 preview index를 계산하고 있어, 우측 bar도 같은 방식으로 맞추는 편이 안전하다.

관련 코드:

- [frontend/src/features/chat/components/message-item.tsx](/home/hskim/work/llm-agent-v2/frontend/src/features/chat/components/message-item.tsx)

### 코드 분석 결과 (확인됨)

`RetrievedDocsContent` (layout/index.tsx:1238-1265):

```typescript
// 문제 1: previewImages가 문서당 1개만 생성 (expanded_page_urls 무시)
const previewImages = docs.map((doc) => ({
  url: doc.page_image_url,  // ← 첫 페이지 URL만 사용
  // expanded_page_urls는 여기서 사용되지 않음
}));

// 문제 2: handleImageClick이 pageIndex를 무시
const handleImageClick = (docIndex: number, _pageIndex: number) => {
  setPreviewIndex(docIndex);  // ← _pageIndex 무시, 항상 문서 대표 페이지
};
```

**수정 방향**: `previewImages`를 페이지 단위로 flat하게 풀고, `handleImageClick`에서 `docIndex + pageIdx`를 올바른 global index로 매핑

```typescript
// 수정안
const previewImages = docs.flatMap((doc) => {
  const urls = doc.expanded_page_urls?.filter(Boolean) ||
               (doc.page_image_url ? [doc.page_image_url] : []);
  const pages = doc.expanded_pages || (doc.page ? [doc.page] : []);
  return urls.map((url, i) => ({
    url, title: displayTitle, page: pages[i], docId: doc.id,
  }));
});

// docIndex → global previewIndex 매핑 테이블 필요
```

**채팅 본문(message-item.tsx)**은 이미 페이지 단위 preview index를 올바르게 계산하고 있어, 이 로직을 참고하면 됨.

## 2. 질문 입력 UX 단순화

### 변경 목표

현재:

- 언어
- 장비모델
- equipment_id
- 절차/이슈

를 질문 후 확인받는다.

변경 후:

- 언어는 묻지 않는다.
- 채팅 입력창 바로 위에서 `절차검색` / `이슈검색`을 사용자가 먼저 선택한다.
- 선택하지 않으면 질문 전송을 막는다.

### 결정사항

#### 2-1. 언어 선택 제거

- 사용자가 질문한 언어로 답변하는 것을 기본값으로 한다.
- 별도 언어 선택 단계는 제거한다.

### 구현 방향

- `target_language`는 guided selection 입력값이 아니라, 질문 언어 감지 결과를 사용한다.
- 질문 언어가 혼합되거나 불명확하면 다음 우선순위를 사용한다.
  1. 사용자 질문 언어 감지 결과
  2. 직전 턴의 답변 언어
  3. 시스템 기본값 `ko`

### 제안

- 언어를 완전히 숨기되, metadata에는 계속 기록하는 것이 좋다.
- 그래야 평가와 디버깅이 가능하다.

#### 2-2. task_mode를 입력창 위 radio로 이동

- 입력창 위에 radio button 배치:
  - `절차검색`
  - `이슈검색`
- 둘 중 하나를 선택하지 않으면 `질문 전송 비활성화`

### 구현 방향

- 기존 guided selection의 `task` 단계는 제거하거나 우회한다.
- `task_mode`는 질문 전 authoritative 값으로 설정한다.
- 백엔드는 기존처럼 이 값을 기준으로 retrieval scope와 answer policy를 결정한다.

### 제안

- 현재 턴뿐 아니라 같은 세션 동안 마지막 선택을 유지하는 것이 좋다.
- 즉:
  - 사용자가 `절차검색`을 선택하면 다음 질문도 기본 유지
  - 사용자가 바꾸고 싶을 때만 radio 변경
- 이 방식이 매 질문마다 다시 선택하는 것보다 UX가 낫다.

#### 2-3. 장비모델 / equipment_id 질문은 유지하되 조건부로 단순화

사용자 요청에는 언어 제거와 task_mode 이동이 핵심이지만, 실제로는 장비모델/equipment_id까지 매번 물으면 여전히 복잡할 수 있다.

### 추가 제안

다음 순서로 단순화하는 것이 좋다.

1. `task_mode`는 사전 선택
2. 질문 입력
3. auto-parse 결과가 충분히 명확하면 바로 검색
4. 장비모델/equipment_id가 불명확할 때만 확인 질문

즉, 장기적으로는 `항상 확인`보다 `불확실할 때만 확인`이 더 낫다.

이번 작업 범위에서는 우선 언어 제거 + task_mode radio 이동을 먼저 한다.

## 3. 절차검색 답변을 SOP의 Work Procedure / Workflow 중심으로 고정

### 문제

- 절차검색인데도 답변이 일반 설명, 배경 설명, 주변 정보 중심으로 나올 수 있다.
- 사용자는 실제 작업 순서를 가장 먼저 원한다.

### 목표

- SOP 문서를 찾았을 때 답변 중심을 다음 순서로 고정한다.
  1. `Work Procedure`
  2. `Workflow`
  3. 필요한 사전 조건 / 주의사항 / 후속 확인

### 구현 방향

- 절차검색(`task_mode=sop`)에서 SOP 문서가 검색되면:
  - `work procedure`
  - `workflow`
  - 이와 유사한 절차성 헤더
  를 우선적으로 인용/요약하도록 프롬프트를 강화한다.
- 절차성 섹션이 없을 경우에만 다른 relevant section을 보조적으로 사용한다.

### 답변 우선순위 규칙 제안

1. 작업 순서가 있는 문서면 순서형 절차를 먼저 제시
2. workflow 다이어그램/플로우가 있으면 단계 순서로 풀어서 설명
3. 사양값, torque, 주의사항은 절차 뒤에 배치
4. 배경 설명은 맨 마지막 또는 생략

### 추가 제안

- retrieval 후 answer 단계에서 문서 전체를 뭉뚱그려 넘기기보다, 절차성 헤더가 포함된 chunk/page를 우선 정렬하는 보정이 있으면 더 안정적이다.
- 즉, 프롬프트 개선만이 아니라 `답변용 ref ordering`도 같이 보는 것이 좋다.

## 4. 절차 답변 양식 고정

### 문제

- 현재 절차 답변은 번호 목록이 나오더라도 구조가 들쑥날쑥하다.
- 어떤 답변은 요약 위주, 어떤 답변은 장황한 서술 위주다.

### 목표

- 절차검색 답변의 형식을 일정하게 만든다.
- 사용자가 항상 같은 위치에서 같은 종류의 정보를 찾을 수 있어야 한다.

### 권장 답변 포맷

절차검색 답변은 아래 순서를 기본 템플릿으로 고정한다.

#### 제안 포맷

1. `작업 목적`
2. `작업 전 확인`
3. `작업 절차`
4. `작업 후 확인`
5. `주의사항`
6. `참고 문서`

예시:

```text
작업 목적
- EFEM PIO SENSOR BOARD 교체 절차 안내

작업 전 확인
- 설비 정지 상태 확인
- 필요한 공구/부품 확인

작업 절차
1. ...
2. ...
3. ...

작업 후 확인
- 전원 인가 후 상태 확인
- 관련 알람 및 통신 상태 확인

주의사항
- ...

참고 문서
- [1] ...
- [2] ...
```

### 구현 방향

- `setup_ans_v2.yaml`를 이 형식에 맞게 강화한다.
- 답변 프롬프트에 다음을 명시한다.
  - 항상 같은 섹션 순서를 사용
  - 절차가 부족하면 부족하다고 말할 것
  - REFS에 없는 절차를 생성하지 말 것

### 추가 제안

- `작업 절차`는 반드시 번호 목록으로 고정하는 것이 좋다.
- `작업 전 확인`, `작업 후 확인`, `주의사항`은 bullet로 고정한다.
- 이렇게 하면 사용자가 절차와 보조 정보를 빠르게 구분할 수 있다.

## 추천 아키텍처

```text
사용자
  → task_mode radio 선택 (절차검색 / 이슈검색)
  → 질문 입력
  → auto_parse

      - 이슈검색: 사례 요약 중심
  → 참고 문서 보기
      - 기본 fit-to-page
```

## 리스크

### 1. 언어 자동 결정 실패

- 혼합 언어 입력이나 짧은 질문에서 오판 가능

대응:

- 불명확할 때는 직전 답변 언어 또는 `ko` fallback

### 2. task_mode radio와 기존 guided selection 충돌

- 현재는 guided selection에도 task가 존재함

대응:

- UI 기준 authoritative source를 하나로 정해야 함
- 권장: 입력창 위 radio를 authoritative로 사용

### 3. 프롬프트 고정 시 답변이 너무 경직될 수 있음

- 모든 절차 질의에 동일 양식을 강제하면, 문서가 빈약한 경우 어색할 수 있음

대응:

- 섹션은 고정하되, 없는 항목은 "문서에서 확인되지 않음"으로 출력

## 추가 제안

사용자 요청 범위 밖이지만, 같이 고려할 가치가 있는 항목이다.

### 제안 1. task_mode는 세션 동안 유지

- 매 질문마다 다시 선택하지 않고, 사용자가 바꿀 때만 변경
- 실제 사용성 측면에서 가장 자연스럽다.

### 제안 2. 절차검색일 때 `절차 답변 카드` 스타일 사용

- 일반 답변 본문과 다르게, 절차검색은 구조화된 카드 형태로 보이게 하면 가독성이 더 좋아진다.
- 예:
  - 작업 전 확인
  - 작업 절차
  - 작업 후 확인
  - 참고 문서

### 제안 3. SOP 문서의 절차성 헤더 우선 정렬

- answer prompt만 바꾸는 것보다, retrieval 결과를 `work procedure` / `workflow` 헤더 포함 여부로 한 번 더 정렬하면 더 안정적이다.
