# 2026-03-10 122B MoE 모델 성능 저하 분석

## 상황

20B dense 모델에서 `qwen3.5:122b-a10b-q4_K_M` (122B MoE, Q4_K_M 양자화)로 교체 후, 답변 품질이 오히려 심각하게 저하됨.

- 질문과 무관한 할루시네이션 (존재하지 않는 URL, 엉뚱한 내용)
- 동일 패턴의 반복 출력 (repeat_penalty=1.3으로도 제어 실패)
- 프롬프트 지시 무시 (REFS 인용 미준수, 언어 혼합)
- judge가 이를 "faithful"로 통과

## 재현 케이스

질문: `"마노미터 수치에 이상이 발생했어"`
task_mode: `issue`, route: `setup` (LLM 분류), device: `SUPRA Vplus`

### 관찰된 증상

| 증상 | 상세 |
|------|------|
| URL 할루시네이션 | `digikey.com`, `techcrunch.naver.com`, `nerdfusion.org`, `github.com/skotok` 등 존재하지 않는 URL 생성 |
| 내용 할루시네이션 | "LP1과 PM3의 **임금 관리** 문제" — 반도체 장비 문맥에 "임금 관리"는 완전한 환각 |
| 반복 루프 | 단계 1~11+ 동일 패턴 무한 반복 (repeat_penalty=1.3, repeat_last_n=256으로도 미제어) |
| 언어 혼합 | 한국어 답변 중 중국어("参考文献") 혼재 |
| 프롬프트 무시 | "REFS에서 인용하세요" 지시 완전 무시, 자체 내용 생성 |
| judge 통과 | `faithful: true`로 판정 — 할루시네이션이 포함된 답변을 승인 |

## 원인 분석

### 1. Retrieval (검색) — 관련 문서 0건

질문 "마노미터 수치에 이상이 발생했어"에 대해 검색된 19개 문서 중 마노미터 관련 문서 **0건**.

| 검색 결과 | 내용 |
|-----------|------|
| 40146973 | Mapping Arm 감도 수치 변경 |
| 40159486 | RFID Read / CTC GUI 인식 불가 |
| 40129396 | LP1 Mapping Arm Open Alarm |
| 40139383 | TCB 탈착 |
| ... | 전부 마노미터와 무관 |

- guardrail이 대부분 제거: `guardrail_final_count: 1` (19개 중 1개만 통과)
- `mq_used: false` — multi-query 확장 미사용
- `40120655`에 "Mano Meter" 언급이 있으나 CDA Leak 내용이고 guardrail에서 제거됨

**결론**: "마노미터"라는 키워드가 인덱스 문서에 희소하여 BM25/dense 검색 모두 매칭 실패.

### 2. Model (LLM) — 122B Q4 MoE의 구조적 한계

#### 20B dense vs 122B Q4 MoE 비교

| 항목 | 20B dense | 122B MoE (q4_K_M) |
|------|-----------|-------------------|
| 전체 파라미터 | 20B | 122B |
| 활성 파라미터 | 20B (전체) | **10B** (a10b) |
| 양자화 | 없음 또는 Q8 | **Q4_K_M** (4bit) |
| 실질 추론 능력 | 20B 수준 | **10B 이하** (양자화 손실 포함) |
| instruction following | 양호 | 저하 |
| REFS 없을 때 행동 | "관련 문서를 찾지 못했습니다" | **할루시네이션으로 공간 채움** |
| 반복 제어 | 정상 | repeat_penalty=1.3으로도 미제어 |

#### 핵심 원인

1. **MoE 활성 파라미터 10B**: 122B 전체 중 추론 시 10B만 활성화 → 실질 성능이 dense 20B보다 낮을 수 있음
2. **Q4_K_M 양자화 손실**: 4bit 양자화가 instruction following, 프롬프트 준수 능력을 크게 저하
3. **빈 REFS에서의 행동 차이**: 검색 결과가 부족할 때 20B는 "없다"고 응답했지만, 122B Q4는 환각으로 빈 공간을 채움
4. **반복 제어 실패**: MoE 구조에서 repeat_penalty가 효과적으로 작동하지 않음 (활성 expert가 변경되면서 penalty 회피 가능성)

### 3. Route/Judge — 잘못된 경로 + 평가 실패

```
task_mode: "issue"     ← 사용자가 이슈조회 선택
route: "setup"         ← LLM이 setup으로 분류 (불일치)
```

- **route/task_mode 불일치**: 이슈조회 의도인데 setup 프롬프트로 "단계별 절차" 형식 생성 시도
- **judge `faithful: true`**: URL 할루시네이션, 환각 내용을 신뢰성 있다고 판정
  - judge가 issues를 감지했으나 (`"LP1와 PM3의 잘여 객관적 정보가 부족함"`) faithful을 true로 판정
  - judge 프롬프트가 URL 검증, 할루시네이션 감지에 부적절

### 실패 흐름 요약

```
검색 실패 (관련 문서 0건)
    ↓
모델이 빈 REFS로 답변 시도
    ↓
122B Q4 MoE가 hallucination으로 공간 채움 (URL, 내용 모두 조작)
    ↓
잘못된 route("setup")로 단계별 형식 강제
    ↓
judge가 이를 "faithful"로 통과
    ↓
repeat_penalty로도 제어 불가한 반복 루프
    ↓
사용자에게 쓰레기 답변 노출
```

## 대응 방안

### 추가 진단 (Model / Prompt / Retrieval 분리)

이번 케이스는 "모델만 나빠졌다"로 단정하기 어렵고, **Model + Prompt/Flow + Retrieval 품질**이 동시에 겹친 복합 장애로 보는 것이 타당하다.

### A. Model 축 (실제 런타임 확인)

- 현재 런타임 LLM은 `vllm(gpt-oss-20b)`가 아니라 `ollama(qwen3.5:122b-a10b-q4_K_M)`가 사용 중.
  - `.env`: `RAG_LLM_METHOD=ollama`, `OLLAMA_MODEL_NAME=qwen3.5:122b-a10b-q4_K_M`
  - `get_default_llm()`는 `rag_settings.llm_method` 기준으로 분기
- 즉, "20B 대비 품질 저하" 체감은 실제로 **Q4 양자화 122B MoE 경로**에서 발생한 것으로 해석 가능.
- 이 모델은 REFS 품질이 낮을 때 환각으로 빈 공간을 채우는 경향이 두드러짐.

### B. Prompt/Flow 축 (분기 불일치)

- 관찰값: `task_mode=issue`인데 최종 `route=setup`.
- 현재 코드에서 `route_node`는 `task_mode`를 고려하지 않고 LLM 분류만 수행함.
- 그 결과 이슈조회 의도여도 setup 프롬프트가 선택될 수 있고, 답변 형식이 의도와 어긋남.
- 추가 확인: 현재 설정에서 `prompt_spec_version` 기본값이 `v1`로 읽히고 있음.
  - v2 프롬프트/분기 개선을 작성했더라도, 실제 런타임이 v1이면 개선이 체감되지 않을 수 있음.

### C. Retrieval 축 (질문-문서 불일치)

- 질문은 "마노미터 수치 이상"인데 상위 검색 결과는 mapping/RFID/door/alarm 등 **저연관 myservice 문서가 다수**.
- `issue` 범위가 `myservice/gcb/ts` 전체로 넓고, 단일 질의(`mq_used=false`)일 때 노이즈가 커짐.
- 입력 REFS 자체가 저연관이면 모델이 프롬프트를 잘 따라도 답변 품질이 급락함.

### D. Judge 축 (품질 게이트 실패)

- 이번 케이스처럼 URL 환각/무관 내용이 많은데 `faithful=true` 통과.
- judge 프롬프트/후처리가 "URL 환각", "REFS 미근거 인용", "언어 혼합"을 강하게 실패 처리하지 못함.
- 결과적으로 불량 답변이 사용자에게 그대로 노출됨.

### 결론 (원인 비중)

1. **1차 원인**: Retrieval 품질 저하 + 분기 불일치(`task_mode`/`route`)  
2. **2차 원인**: Q4 MoE 모델의 프롬프트 준수/안전성 약화  
3. **증폭 요인**: Judge 게이트 실패

즉, 이번 장애는 모델 단일 이슈가 아니라 파이프라인 정합성 문제다.

### 즉시 조치 (모델 변경 없이)

| 우선순위 | 대응 | 계층 | 상세 |
|----------|------|------|------|
| **P0** | REFS 0건 시 답변 차단 | answer_node | `guardrail_final_count`가 0~1이면 "관련 문서를 찾지 못했습니다" 고정 응답. LLM 호출 자체를 skip |
| **P0** | URL 생성 금지 프롬프트 | 답변 프롬프트 | "절대로 URL을 생성하지 마세요. REFS에 없는 정보를 만들어내지 마세요." 명시적 추가 |
| **P1** | judge 강화 | judge_node | URL 패턴(`http://`, `https://`) 감지 시 자동 `faithful=false`. REFS에 없는 doc_id 인용 시 자동 실패 |
| **P1** | route/task_mode 불일치 해소 | route_node | Phase A 작업 — task_mode가 설정되면 LLM 분류 skip (이미 문서화됨) |
| **P2** | multi-query 강제 사용 | mq_node | 단일 검색어로 결과 부족 시 자동 MQ 확장 (현재 `mq_used: false`) |
| **P2** | 할루시네이션 후처리 | answer_node | 생성된 답변에서 URL 패턴 제거 + REFS 번호 범위 밖 인용 제거 |

### 모델 레벨 대응

| 옵션 | 설명 | 리스크 |

## branching-improvement 작업과의 관계

현재 진행 중인 `agent-branching-improvement` (Phase A) 작업이 이 문제의 일부를 해결한다.

### 해결되는 항목

| 문제 | branching-improvement 작업 | 효과 |
|------|---------------------------|------|
| route/task_mode 불일치 | Task 1: route_node에서 task_mode 우선 적용 | `task_mode=issue`면 route가 `setup`으로 빠지지 않음 |
| issue 프롬프트 품질 | Task 2-3: issue 전용 프롬프트 v2 적용 | 이슈조회에 맞는 답변 형식 생성 |
| prompt_spec_version v1 고정 | Task 4-5: v2 프롬프트 활성화 | 개선된 프롬프트가 실제 런타임에 반영 |

### 해결되지 않는 항목 (branching-improvement 이후 별도 작업 필요)

| 문제 | 필요한 작업 | 우선순위 |
|------|------------|----------|
| REFS 0건 시 할루시네이션 | answer_node에서 guardrail_final_count ≤ 1 시 LLM 호출 skip, 고정 응답 반환 | **P0** |
| URL 할루시네이션 | 답변 프롬프트에 URL 생성 금지 명시 + 후처리에서 URL 패턴 제거 | **P0** |
| judge 품질 게이트 실패 | judge_node에서 URL 패턴 감지 시 자동 `faithful=false`, REFS 미근거 인용 감지 | **P1** |
| 검색 품질 (도메인 용어) | query_expansion에 동의어 사전 추가 (마노미터 ↔ manometer ↔ 압력계) | **P2** |
| MQ 미사용 | 검색 결과 부족 시 자동 multi-query 확장 | **P2** |
| 모델 instruction following | Q4 → Q8 양자화 변경 또는 20B 복원 평가 | **P2** |

### 작업 순서

```
1단계: branching-improvement Phase A 완료 (route 불일치 + 프롬프트 v2)
    ↓
2단계: P0 — REFS 0건 차단 + URL 생성 금지 (모델 변경 없이 즉시 효과)
    ↓
3단계: P1 — judge 강화 (할루시네이션 게이트)
    ↓
4단계: P2 — 검색 개선 + 모델 평가
```

branching-improvement만으로는 **Retrieval 실패 → 빈 REFS → 할루시네이션** 경로를 차단할 수 없다. 2단계(P0)가 이 경로의 핵심 방어선이다.

## 구현 우선순위

1. **REFS 0건 답변 차단** — answer_node에서 guardrail_final_count ≤ 1 시 고정 응답
2. **URL 생성 금지 + judge URL 감지** — 프롬프트 수정 + judge 후처리
3. **route/task_mode 불일치 해소** — Phase A 작업 (이미 계획됨)
4. **모델 양자화 레벨 검토** — Q4 → Q8 또는 20B 복원 평가
5. **검색 동의어 확장** — 도메인 용어 커버리지 개선

## 체크포인트

- [ ] REFS 0건 시 LLM 호출 없이 "관련 문서를 찾지 못했습니다" 응답되는지
- [ ] URL이 포함된 답변이 judge에서 자동 reject되는지
- [ ] task_mode="issue"일 때 route가 고정되어 setup 프롬프트로 빠지지 않는지
- [ ] "마노미터" 검색 시 관련 문서가 1건 이상 반환되는지
